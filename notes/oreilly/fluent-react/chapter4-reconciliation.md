# Reconciliation（調和・リコンシリエーション）

Reactが仮想DOM（望ましいUI状態）を受け取り、実際のDOM環境（ブラウザなど）に反映するプロセス全体のこと。

## 仮想DOMとReact要素

- JSXはコンパイルされて `React.createElement()` 呼び出しになる
- 各React要素は **不変の「望ましい状態」** を表すオブジェクト（実際のUI状態ではない）
- 構造は `type`（要素の種類）と `props`（属性・子要素）のネストされたオブジェクト

### バッチ処理

複数の `setState` 呼び出しを1回のDOM更新にまとめることで、不要なリフロー/再描画を削減する。

```js
// setCount を3回呼んでも DOM は1回だけ更新される (count: 0 → 3)
setCount(c => c + 1)
setCount(c => c + 1)
setCount(c => c + 1)
```

ブラウザの `DocumentFragment` と同様に、複数のvDOM変更をステージングしてから一括コミットするイメージ。

## スタック・リコンシラー（旧）

LIFO（後入先出）スタックを使った旧来の実装。以下の致命的な欠点があった：

- **優先度付け不可** — 受信順に処理するため、低優先度の更新が高優先度の更新をブロックする
- **中断・キャンセル不可** — 一度開始した処理を止められない
- 結果としてUIのジャンク（カクつき）やテキスト入力の遅延が発生

## Fiber・リコンシラー（現在）

スタック・リコンシラーの問題を解決するために開発された新アーキテクチャ。

### React要素 vs Fiberノード

| | React要素 | Fiberノード |
|---|---|---|
| 寿命 | 一時的（エフェメラル） | 長寿命 |
| 状態 | ステートレス | ステートフル |
| 表すもの | 望ましい状態 | 実際のコンポーネントツリー |

Fiberノードは Props・State・子コンポーネント情報に加え、ツリー内の位置情報（親・子・兄弟の参照）も保持する。

### ダブルバッファリング

グラフィックスのちらつき防止技術を応用。

- `current` ツリー：現在画面に表示されているもの
- `workInProgress` ツリー：次の状態を構築中のもの
- 準備できたら `commitRoot` でポインタを切り替えて反映

### 2段階の処理フェーズ

#### レンダリングフェーズ（割り込み可能）

- **`beginWork` 関数** — ツリーを再帰的に下りながら「更新が必要か」フラグを設定
- **`completeWork` 関数** — メモリ上（オフスクリーン）でDOMツリーを構築
- **`renderLanes`** — ビットマスクで更新優先度を管理。ユーザー操作など重要な更新を優先
- **タイムスライス** — スケジューラーが約5msごとにメインスレッドを解放し、高優先度更新への割り込みを許可

#### コミットフェーズ（割り込み不可）

1. **ミューテーションフェーズ** — 仮想DOMの変更を実DOMに適用、不要ノードを削除
2. **レイアウトフェーズ** — ブラウザペイント前に新しいレイアウトを計算

### エフェクトの種類

| エフェクト | タイミング | フック |
|---|---|---|
| 配置 | 新規追加時 | — |
| 更新 | 更新時 | — |
| 削除 | 削除時 | — |
| レイアウト | ペイント**前** | `useLayoutEffect` |
| パッシブ | ペイント**後** | `useEffect` |

### FiberRootNode

`current` と `workInProgress` の2ツリーを管理する中心データ構造。`commitRoot` 関数がworkInProgressをcurrentに切り替え、以降の更新はすべてこの新ツリーを基準に実行される。

## 実務で意識すべき注意点・コーディングガイド

- `key` には安定したIDを使う（indexは不可）— Reconciliationがノードを同定するため、indexだと順序変更時に全再マウントが起きる
- `setState` で前の値を使うときは関数形式 `setState(s => s + 1)` — バッチ処理でstale valueになるのを避ける
- `useEffect` の依存配列は正確に書く — 不完全だとstale closureになる
- `useLayoutEffect` はDOM測定・スクロール位置調整のみ — ペイント前ブロッキングのため乱用しない
- `useMemo` / `useCallback` / `React.memo` は計測してから使う — 参照比較コストが計算コストを上回ることもある
- オブジェクト・配列をpropsに直書きしない — レンダリングごとに新しい参照になりchildが必ずre-renderする
- 状態は使う場所のできるだけ近くに置く（State Colocation）— 上位に持ちすぎると不要な再レンダリングが広がる
- `startTransition` で低優先度更新をマークする — ユーザー入力（高）と検索結果更新（低）を分離できる
- `useEffect` のクリーンアップを必ず書く — Strict Modeで二重実行されるため、漏れが開発時に検出される
- Context は変更頻度の高い値に使わない — Contextの値が変わると購読している全コンポーネントが再レンダリングされる

## まとめ

```
JSX
 ↓ コンパイル
React要素ツリー（仮想DOM）
 ↓ バッチ処理・差分計算
workInProgress Fiberツリー（オフスクリーン構築）
 ↓ コミットフェーズ
実DOM更新（ミューテーション → レイアウト → ペイント）
```

Fiberアーキテクチャにより、優先度付き・中断可能なレンダリングが実現された。
