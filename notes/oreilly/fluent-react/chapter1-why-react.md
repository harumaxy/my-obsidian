# Chapter 1: なぜ React なのか

> 本書の目的は React の**使い方**ではなく、**どう機能するか**を理解すること。ライブラリ開発者レベルの深い理解を目指す。

## React が生まれた背景

- Web アプリが静的ページから動的 UI に進化し、ページ全体リロード不要の瞬時更新が求められた
- React 以前の 3 大課題:
  - **パフォーマンス**: DOM 操作がリフロー/リドローを引き起こしボトルネックに
  - **信頼性**: 複数の開発者が関わる際の UI 状態の同期が困難
  - **セキュリティ**: XSS / CSRF 対策が開発者任せ

## フロントエンドフレームワークの歴史

### jQuery 時代

- DOM を直接操作するため振る舞いが予測不可能
- ブラウザ依存が強くテストが困難
- モダンブラウザが `querySelector` などをネイティブ実装したことで冗長化・衰退

**Vanilla JS / jQuery の限界**:
- ボタン 1 つ実装するだけでも、要素取得 → イベント登録 → 状態管理 → 通信 → エラーハンドリングをすべて手動で行う必要があった
- 「いいね」ボタン 1 つでも「通常・保留中・成功・失敗」の状態を管理し、各状態で UI 更新が必要
- `addEventListener` / `removeEventListener` のタイミング管理が複雑でメモリリークのリスクがある
- JavaScript の変数と DOM の状態が分離していて「どちらが正しいか」が曖昧

### Backbone.js

- **革新**: Model（データ）と View（UI）を分離する MVC パターンを導入し、テスト・再利用性が向上
- **限界**:
  - ボイラープレートコードが多い
  - 双方向データバインディングなし
  - イベント連鎖のデバッグが困難
  - コンポーネント入れ子機能の欠如

### KnockoutJS（2010年頃）

- **MVVM パターン**（Model / View / ViewModel）でデータバインディングを自動化
- ViewModel が特定の View を意識しない設計でデカップリング度が高い
- `data-bind` 属性でバインディングするが、定型コードが多く ViewModel が肥大化しやすい

### AngularJS（Google, 2010年）

- **双方向データバインディング**: モデル変更 → ビュー自動更新（逆も然り）で手動 DOM 操作が不要に
- **依存性注入（DI）とモジュール化**: コンポーネントの独立開発・テスト・保守を可能に
- **問題点**:
  - ダイジェスト・サイクル（変更検出）が複雑なアプリで遅延を引き起こす
  - Angular 2+ への移行は後方互換性なし・TypeScript 必須で大規模な書き直しが必要
  - → コミュニティ分裂・React 台頭のきっかけに

## React の設計思想

### 誕生の背景

- Facebook 内で Jordan Walke が「ページの最小部分を丸ごと置き換える」という急進的なアイデアを提案
- 初期は批判されたが Netflix・Airbnb・NYTimes などに採用され Web UI のデファクトスタンダードへ

### 核となる概念

| 概念 | 説明 |
|------|------|
| **コンポーネント** | UI を「props を受け取り UI を返す関数」として捉える |
| **一方向データフロー** | データ変化が追跡しやすく、予測可能 |
| **仮想 DOM（Virtual DOM）** | 軽量な JS オブジェクトで DOM を表現し差分のみ更新 → パフォーマンス向上 |
| **宣言的 UI** | 「どう更新するか」ではなく「UI がどう見えるべきか」を記述するだけ |
| **不変性（Immutability）** | 状態を直接変更せず新しいオブジェクトとして返す → 変更追跡が容易 |
| **コンポジション** | シンプルなコンポーネントを組み合わせて複雑な UI を構築 |
| **キーイング（Keying）** | React がコンポーネントを経時的に追跡し、メモ化・バッチ化などの最適化を実現 |

### Flux アーキテクチャ

```
Action → Dispatcher → Store → View
```

- 一方向のデータフローでアプリケーションの動作を予測可能に
- 単一の信頼できる情報源・テスト可能性の向上・関心事の明確な分離（SoC）

### React の価値提案

- コンポーネントベースアーキテクチャ
- 宣言型プログラミング
- Virtual DOM によるパフォーマンス最適化
- 広大なエコシステム
- プラットフォーム非依存（Web / Native）
- Meta のバックアップ
