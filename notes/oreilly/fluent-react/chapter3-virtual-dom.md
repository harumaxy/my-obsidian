# Fluent React 第3章: 仮想DOM

## 仮想DOMとは

仮想DOMはJavaScriptオブジェクトで表現されたHTMLドキュメントの軽量な記述表現。実DOMと異なりプレーンなJSオブジェクトで構成され、JavaScriptエンジン内で高速に操作できる。

UIに変更が加わるとまず仮想DOMが更新され、その後リコンシリエーション（調停）によって実DOMが更新される。

## 実DOMのパフォーマンス問題

### リアルDOM操作のコスト

- DOM変更時にブラウザはレイアウト再計算（リフロー）と再描画処理が必須
- 大規模・複雑なページほどコストが高く、低電力デバイスで深刻
- `offsetWidth` の読み取りのような一見簡単な操作でもリフローが発生する

- また、DOMは全体的に複雑で、揮発性が高く、状態管理に悩まされるので、リアルDOMを使いこなす難易度は並ではない
- DOMにはブラウザごとでモデル化が異なっており、クロスブラウザ対応が必要な可能性もある(特定DOM要素、属性があるブラウザでしかサポートされてないなど)

### DOM要素選択の計算量

| メソッド | 計算量 | 理由 |
|---|---|---|
| `getElementById` | O(1) | IDはハッシュテーブルで管理 |
| `querySelector` | O(n) | CSSセレクタでDOMツリー全体を走査 |

### パフォーマンス最適化テクニック（仮想DOMなし）

- `getBoundingClientRect()` でレイアウトの読み取り・書き込みをバッチ処理
- 読み取りと書き込みを分離してレイアウトスラッシングを最小化
- `requestAnimationFrame` で遅延処理

## 仮想DOMの仕組み

### ドキュメントフラグメント

実DOMに影響なく複数の変更を一時的にまとめるコンテナ。最終的に1回のリフロー・再描画で実DOMに追加される。仮想DOMはこの概念を拡張したもの。

### 更新の3段階

1. 状態変更時に新しい仮想DOMツリーを作成
2. 前のツリーと差分比較（diffing）して変更箇所を特定
3. 最小限の変更をバッチ化して実DOMに適用

## SyntheticEvent システム

Reactのクロスブラウザ互換性対策。ネイティブイベントをラップして統一インターフェースを提供する。

- ブラウザごとの不整合（`event.target` vs `event.srcElement` など）を解消
- ルートレベルでイベントをリッスンするイベント委譲を採用
- `onChange` など入力要素間の振る舞いの違いを統一

## React要素の内部構造

`React.createElement()` が返すプレーンなJavaScriptオブジェクト。

```js
{
  $$typeof: Symbol(react.element),  // 型マーカー
  type: "div",                      // HTML要素またはコンポーネント関数
  key: null,                        // リスト要素の識別子
  ref: null,                        // DOM直接参照
  props: {                          // 属性・子要素
    className: "foo",
    children: [...]
  },
  _owner: ...,  // 内部用：作成元コンポーネント（開発モードのみ）
  _store: ...,  // 内部用：バリデーション等のメタデータ
}
```

- `$$typeof`: Symbol でReact要素の種類を識別する型マーカー（後述）
- `type`: 文字列（`"div"` 等）またはカスタムコンポーネント関数
- `_owner` / `_store`: 内部実装の詳細、アプリコードから直接使用しない

> `React.createElement` はメモリ上の仮想要素を作成し、`document.createElement` はDOMノードを作成する。

### `$$typeof` の種類

Symbolを使う理由：JSONから復元できないため、サーバーから注入されたJSONをReact要素として誤認させるXSSを防ぐセキュリティ上の仕掛け。

| Symbol | 役割 | 主な用途 |
|---|---|---|
| `Symbol(react.element)` | 通常のReact要素 | JSX全般・`React.createElement()` |
| `Symbol(react.fragment)` | DOMノードなしのグループ化 | `<>...</>` / `<React.Fragment>` |
| `Symbol(react.portal)` | 別DOMノードへのレンダリング | モーダル・ツールチップ（`z-index`/`overflow`の影響回避） |
| `Symbol(react.profiler)` | レンダリング時間計測 | パフォーマンス調査・開発時のみ推奨 |
| `Symbol(react.provider)` | Contextの値注入 | テーマ・認証情報などのツリー伝播 |

Portalは**DOMツリー上は別の場所**にレンダリングされるが、**仮想DOMツリー上の親子関係は維持**されるためイベントバブリングは通常通り親に伝わる。

## 差分アルゴリズム（Diffing）

### 最適化の工夫

- **ルートノード判定**: ルートレベルのノードが変わった場合はサブツリーごと置き換え
- **属性更新**: 変更された属性のみ更新
- **子ノード順序**: 並び替えを効率的に検出
- **`key` プロップ**: リスト要素の同一性を追跡し、不要な再生成を防ぐ

### 不要な再レンダリング問題

親コンポーネントの状態変更時に、propsが変わらない子コンポーネントまで再レンダリングされてしまう。

**対策**: `React.memo`、`useMemo`、`useCallback` を適切に活用して再レンダリングを制御する。

## 用語集

| 用語（日本語） | 英語表記 | 概要 |
|---|---|---|
| リフロー | Reflow / Layout | ブラウザが各要素の位置・サイズを計算し直す処理。DOM変更やレイアウト依存プロパティ読み取りで発生し、非常にコストが高い |
| 再描画 | Repaint | リフロー後にブラウザが画面にピクセルを描き直す処理。色変更など位置に影響しない変更はリフローを伴わない |
| レイアウトスラッシング | Layout Thrashing | 読み取り（`offsetWidth`など）と書き込み（スタイル変更）を交互に繰り返してリフローが連鎖発生する状態 |
| 調停 / リコンシリエーション | Reconciliation | Reactが仮想DOMの新旧ツリーを比較して実DOMへの最小限の変更を計算・適用するプロセス全体 |
| 差分アルゴリズム | Diffing | リコンシリエーション内で行われる新旧仮想DOMツリーのノード比較処理。変わった箇所だけ実DOMに反映する |
| ファイバー | Fiber | React 16以降のリコンシリエーション実装。処理を中断・再開可能な単位に分割しUIの応答性を維持する |
| イベント委譲 | Event Delegation | 個々の要素ではなくルート要素1箇所でイベントをまとめてリッスンする手法。バブリングを利用する |
| 合成イベント | SyntheticEvent | Reactがネイティブイベントをラップした統一インターフェース。ブラウザ間のAPI差異を吸収する |
| ドキュメントフラグメント | DocumentFragment | 実DOMに接続されていない仮のコンテナ。複数のDOM変更をまとめて一度だけ実DOMに追加しリフローを1回に抑える |

## まとめ

Reactは仮想DOMを中間層として差分アルゴリズムで最小限の実DOM操作に絞り込むことで、開発者がDOMの複雑さを意識せず高パフォーマンスなUIを構築できる仕組みを提供している。「まとめてバッチ処理する」という設計哲学がSyntheticEvent・ドキュメントフラグメント・差分アルゴリズム全体を貫いている。
