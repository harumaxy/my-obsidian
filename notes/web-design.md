## Designing for iOS

> https://developer.apple.com/design/human-interface-guidelines/designing-for-ios

### iPhoneの基本特性

| 特性 | 内容 |
|---|---|
| ディスプレイ | 中サイズの高解像度 |
| エルゴノミクス | 片手/両手持ち、縦横切り替え、視認距離1〜2フィート |
| 入力 | マルチタッチ、バーチャルキーボード、音声、ジャイロ/加速度計 |
| 使われ方 | 数分〜1時間超まで幅広く、複数アプリ同時起動が普通 |

### ベストプラクティス

1. **コントロールを絞る** — 画面上の操作要素を減らし、主要タスクに集中させる
2. **外観変更に対応** — ダークモード・ダイナミックタイプ・向き変更をシームレスに
3. **持ち方を考慮した配置** — 画面中央〜下部にコントロールを置くと操作しやすい
4. **プラットフォーム機能を活用** — 位置情報・生体認証・Apple Payなどをユーザー許可のもとで統合

### メモ

- **ダイナミックタイプ** — ユーザーがOS設定で変更した文字サイズにアプリが追従する仕組み。固定pxではなくremで組む発想と同じ。高さ固定のコンテナに文字を詰め込まない
- **許可ダイアログ（位置情報・カメラ・生体認証など）**
  - 起動直後に全部求めるのはアンチパターン。機能を使う文脈でその場で求める
  - 拒否された場合：機能を完全に殺さない、何が使えなくなるか説明する、再ダイアログは出せないので設定アプリへの誘導UIを用意する
  - Webの `getUserMedia()` や通知APIでも同じ設計原則が適用される

## Designing for macOS

> https://developer.apple.com/design/human-interface-guidelines/designing-for-macos

### Macの基本特性

| 特性 | 内容 |
|---|---|
| ディスプレイ | 大型・高解像度。外部ディスプレイで作業スペース拡張も可 |
| エルゴノミクス | デスク/テーブル固定。視認距離1〜3フィート（iOSより遠い） |
| 入力 | 物理キーボード、ポインティングデバイス、ゲームコントローラ、Siri |
| 使われ方 | 数分の短作業〜数時間の深い集中まで。複数アプリ同時起動が普通 |

### macOS固有のシステム機能

- **メニューバー** — アプリの全コマンドに素早くアクセス
- **ファイル管理** — Finder統合、ドラッグ&ドロップ操作
- **フルスクリーン表示** — 邪魔のない没入コンテキスト
- **Dockメニュー** — アプリの状態やクイックアクションを表示

### ベストプラクティス

1. **大型ディスプレイを活かす** — 階層を浅くし、モーダルを減らし、情報密度を高めて一画面に多くのコンテンツを
2. **ウィンドウ管理に対応** — リサイズ・最小化・移動・フルスクリーンをすべて自然に扱えるように
3. **メニューバーを充実させる** — アプリ内の操作はすべてメニューバーからもアクセスできるように
4. **高精度入力を活かす** — マウス/トラックパッドでピクセル単位の操作を可能に
5. **キーボードショートカット対応** — 標準ショートカットを守り、キーボードのみでも操作できるように
6. **パーソナライズをサポート** — ツールバーカスタマイズ、表示色・フォント選択などを許可する

### メモ

- iOSと比べてディスプレイが大きく視認距離も遠いため、**情報密度を上げてモーダルを減らす**方向が正解
- iOS は「モバイル = 絞る」、macOS は「デスクトップ = 広げる」という対比で覚えると整理しやすい
- **メニューバーはmacOS固有の重要UI** — Webやモバイルには存在しない。アプリの全機能をここに集約する慣習
- キーボードショートカットへの対応はmacOSでは必須レベル（iOSでは任意）

## Color

> https://developer.apple.com/design/human-interface-guidelines/color

### カラーの役割

- コミュニケーションの強化（ステータス・フィードバック）
- ブランドイメージの喚起
- 視覚的連続性の提供
- 情報の理解を助ける

### ベストプラクティス

1. **同じカラーを別の意味に使わない** — ブランドカラーをインタラクティブボタンに使ったなら、非インタラクティブなテキストに同色を使わない
2. **ライト・ダーク・高コントラストすべてでテスト** — システムカラーは自動対応済み。カスタムカラーは明暗両バリアント＋コントラスト強化版を用意
3. **さまざまな照明条件でテスト** — 屋外（明るい）では色は暗く落ち着いて見え、暗い環境では明るく彩度高く見える
4. **さまざまなデバイスでテスト** — True Toneディスプレイは環境光に合わせてホワイトポイントを自動調整する
5. **半透明要素の影響を考慮** — ツールバーなど半透明UIの背後に置いたカラーは見え方が変わる
6. **カラーピッカーはシステム提供のものを使う** — 一貫したUXとユーザーのカラーセット保存が可能

### インクルーシブなカラー（アクセシビリティ）

- **カラーだけで情報を伝えない** — テキストラベルやグリフ形状で補完する（色覚障がい対応）
- **コントラスト不足に注意** — アイコン・テキストが背景に溶け込まないようにする
- **文化によるカラーの意味の違いに注意** — 例：緑＝好調（英語圏の株アプリ）、赤＝好調（中国語圏の株アプリ）

### システムカラー

- **ハードコーディング禁止** — 実際の値はリリースごとに変動する可能性がある。`Color` APIを使う
- ダイナミックシステムカラーは**目的（セマンティクス）で定義**されている
- 意味を勝手に再定義しない（例：`separator` をテキスト色に使うのはNG）

#### iOSの主要なダイナミックカラー（前景コンテンツ）

| カラー名 | 用途 |
|---|---|
| `label` | メインテキスト |
| `secondaryLabel` | 補足テキスト |
| `tertiaryLabel` | 三次テキスト |
| `placeholderText` | プレースホルダー |
| `separator` | 区切り線（背景透過） |
| `opaqueSeparator` | 区切り線（背景非透過） |
| `link` | リンクテキスト |

### Liquid Glass とカラー（iOS 26 / macOS 26〜）

- デフォルトでは**背後のコンテンツのカラーを反映**（固有色なし）
- 着色ガラス（ステンドグラス風）にすることも可能 — プライマリCTAなど**強調が必要な要素に限定**して使う
- ツールバー・タブバーのシンボル・テキストはモノクロ配色に従い、背景の明暗に応じて自動反転
- **複数のコントロール背景に一度にカラーを付けない** — くどくなりラベルが読みづらくなる

### メモ

- CSSの `prefers-color-scheme` + セマンティックカラー変数に相当するのがシステムカラー
- カラーだけに頼らない設計はWebアクセシビリティ（WCAG）と同じ原則
- **システムカラー最優先、カスタムカラーは差別化のためだけ**がAppleの推奨スタンス

### 補足：システムカラーとは

iOSやmacOSがOS側であらかじめ定義しているカラーセット。**「何に使うか」という目的ベースで名前がついている**のが特徴。

- `label` / `secondaryLabel` / `separator` / `systemBackground` など
- 使うだけでライト/ダーク/ハイコントラストへの対応が**自動**になる
- 値をハードコード（`#FF0000`など）すると、ダークモードで見づらくなるアンチパターンになる

Web で言うと CSS カスタムプロパティ（`var(--color-text-primary)`）を `prefers-color-scheme` で切り替えるのと同じ発想。Appleはその変数セットをOS側で標準提供している。

### 補足：True Tone ディスプレイ

環境光センサーで周囲の光の色温度を検出し、ディスプレイのホワイトポイント（白の色味）を自動調整するAppleの機能。

- 暖色照明の下 → ディスプレイも黄みに寄せる → 紙に近い見え方になり目が疲れにくい
- アプリ側でこの効果を強めたり弱めたりできる（例：写真編集アプリは無効化、読書アプリは強化）

### 補足：マルチプラットフォームでの現実的な対応

システムUIに乗っかるのはネイティブ専用アプリでは有効だが、Android/Windows/Webを含むマルチプラットフォームでは難しい。

**2つの戦略：**
1. **プラットフォームごとにUIを合わせる（ネイティブ準拠）** — 各OSらしいUIになるが、コードが増えてメンテ大変
2. **独自デザインシステムで統一（クロスプラットフォーム準拠）** — Flutter / React Native など。Spotify・Notion・Slackはこっち

HIGは「Appleルールへの準拠」より、**良いUIの普遍的原則を学ぶ教材**として読む使い方が現実的。

### 補足：テストの妥協点

全照明・全デバイスの網羅テストは現実的に無理。**「壊れにくい設計にする」ほうがROIが高い。**

- システムカラーを使う → 照明差はOS側がある程度吸収
- コントラスト比を WCAG AA（4.5:1）以上にする → 多少の環境差があっても読める
- 彩度に頼りすぎない → 色覚・照明差・デバイス差すべてに強くなる
- ライト/ダーク目視確認 + Contrast等のツールでチェック → 最低限これで十分

### 補足：デザイントークン

値そのものではなく「意味」に名前をつける設計思想。色・フォントサイズ・shadow・corner-radiusすべてに適用できる。

```css
/* NG：ハードコード */
color: #1a1a1a;
font-size: 14px;
border-radius: 8px;

/* OK：トークン経由 */
color: var(--color-text-primary);
font-size: var(--text-sm);
border-radius: var(--radius-md);
```

- Tailwind では `tailwind.config.js` でトークン定義 → `text-foreground` `bg-card` `rounded-md` などで使う
- shadcn/ui はこの思想をそのまま採用
- **HIGのシステムカラー/ダイナミックタイプ/角丸もすべて同じ思想** — Appleがトークン定義をOSレベルで提供しているのが本質

### 補足：テキスト折り返しと文字サイズ縮小はアンチパターン

divサイズに収めるために `font-size` を `px` で縮小するのはアンチパターン。

**問題：**
- ダイナミックタイプ（ユーザーのOS設定）を壊す
- アクセシビリティ違反になる
- HIGで明示的にNGとされている「コンテナに文字を詰め込む」設計

**正しい対処：文字がはみ出るのはコンテナ側の問題として解決する**

```css
/* NG：文字を縮める */
font-size: 11px;

/* OK：コンテナを柔軟にする */
height: auto;
min-height: 48px;
```

折り返しの判断：
- 折り返してOK → `white-space: normal` で自然に折り返させる
- 折り返したくない → `text-overflow: ellipsis` で省略する
- どうしても1行に収めたい → `font-size` を縮めるのではなく、文言自体を短くする（設計側で解決）

**根本原則：「文字をコンテナに合わせる」ではなく「コンテナを文字に合わせる」**

### 補足：インクルーシブデザインとは

「障がいの有無・年齢・文化背景に関わらず、すべての人が使えるように設計する」考え方。

| 用語 | ニュアンス |
|---|---|
| バリアフリー | 既存の障壁を後から取り除く。主に身体障がい向け |
| ユニバーサルデザイン | 最初からすべての人向けに作る。7原則で定義 |
| インクルーシブデザイン | 多様性をデザインの駆動力として使うニュアンスが強い |

包含関係：インクルーシブ ⊃ ユニバーサル ⊃ バリアフリー。HIGの文脈ではほぼ同義。

**制約から生まれた設計が全員に便利になる例：**
- 字幕 → 難聴者向けだが、騒がしい場所・言語学習にも使われる
- 音声入力 → 運動障がい向けだが、運転中・料理中にも便利
- ハイコントラスト → 弱視向けだが、屋外の強い日差しでも見やすい

### 補足：ロケールによるカラー切り替えの実装（Tailwind）

株価アプリの「上昇色」が英語圏は緑、中国語圏は赤、のような文化差をコードで扱う方法。

**ポイント：色名（`red`/`green`）ではなく意味（`stock-positive`）でトークンを持つ。**

```js
// tailwind.config.js
colors: {
  'stock-positive': 'var(--color-stock-positive)',
  'stock-negative': 'var(--color-stock-negative)',
}
```

```css
/* globals.css — data属性セレクタでロケールごとに値を差し替え */
[data-locale="en"] {
  --color-stock-positive: #22c55e; /* 緑 */
  --color-stock-negative: #ef4444; /* 赤 */
}
[data-locale="zh"] {
  --color-stock-positive: #ef4444; /* 赤 */
  --color-stock-negative: #22c55e; /* 緑 */
}
```

```tsx
// ルートでロケールをセット
<html data-locale={locale}>

// コンポーネント側はロケールを意識しない
<span className={isPositive ? 'text-stock-positive' : 'text-stock-negative'}>
```

### 補足：CSS属性セレクタ

`[data-locale="en"]` は標準CSSの**属性セレクタ**。

```css
[属性名="値"] { }   /* 完全一致 */
[disabled] { }      /* 属性が存在する要素 */
[type="submit"] { } /* input[type="submit"] など */
```

`data-*` はHTML5のカスタムデータ属性で自由に名前をつけられる。CSSのフックとして使うのが一般的。

Tailwindの `dark:` バリアントも内部的に同じ仕組み（`[data-theme="dark"]` または `prefers-color-scheme` を見ている）。

## Typography（タイポグラフィ）

### 概要

タイポグラフィの選択は、判読可能なテキストの表示・情報階層の伝達・ブランドの表現に役立つ。

---

### 視認性の確保

- 各プラットフォームの**推奨デフォルトサイズ・最小サイズ**に従う

| プラットフォーム | デフォルト | 最小 |
|---|---|---|
| iOS/iPadOS | 17pt | 11pt |
| macOS | 13pt | 10pt |
| tvOS | 29pt | 23pt |
| visionOS | 17pt | 12pt |
| watchOS | 16pt | 12pt |

- **ライトウェイト（Ultralight/Thin/Light）は基本的に避ける** → 小さいサイズで読みにくい
- カスタムフォントでライトウェイトを使う場合は推奨サイズより大きくする

---

### 階層の伝達

- フォントのウェイト・サイズ・カラーで重要度を表現する
- **使う書体の種類は最小限に**（多種混在 → 読みにくく統一感がなくなる）
- テキストサイズが変わっても相対的な階層を保つ
- テキストサイズ拡大時は**重要なコンテンツを優先**して拡大（タブタイトルなど周辺UIは必ずしも拡大しなくてよい）

---

### システムフォント

Appleは2つの書体ファミリーを提供：

| フォント | 種類 | 用途 |
|---|---|---|
| **San Francisco (SF)** | サンセリフ | iOS/macOS/tvOS/visionOS/watchOS のUI全般 |
| **New York (NY)** | セリフ | 記事・読み物コンテンツなど |

- プラットフォームごとのシステムフォント：iOS/macOS/tvOS/visionOS → SF Pro、watchOS → SF Compact
- SFとNYは**可変フォント（Variable Font）形式** → 1ファイルで複数スタイルを補間生成できる
- Ultralightから Black まで幅広いウェイトを持つ
- **SF Symbols** がSFフォントとシームレスに統合 → テキストと同じウェイトのアイコンを自動整列

#### テキストスタイル（Text Style）

システムが定義するタイポグラフィ属性セット（大きなタイトル・タイトル・見出し・本文・脚注・キャプション等）。

- サイズ・ウェイト・レディング（行間）が各スタイルに定義されている
- **内蔵テキストスタイルを使うと、ダイナミックタイプが自動で機能する**

---

### ダイナミックタイプ（Dynamic Type）

iOS/iPadOS/tvOS/visionOS/watchOS のシステム機能。ユーザーがシステム設定でテキストサイズを変更するとアプリ全体に反映される。

- **macOSはダイナミックタイプ非対応**
- カスタムフォントを使う場合も同じ動作を実装する必要がある
- レイアウトはあらゆるフォントサイズに対応させる（テキスト切れを起こさない）
- フォントサイズが大きくなったら列数を減らすなどスタックレイアウトで対応
- SF Symbolsを使えばアイコンもダイナミックタイプに自動追従

---

### visionOS の注意点

- 基本は**2Dテキスト**が望ましい（3Dテキストは読みにくい）
- テキストは**ビルボード表示**（装着者の方向を向く）にする → 横や斜めから読めなくなるのを防ぐ
- バックグラウンドとのコントラストを最大化する（デフォルト白テキスト推奨）

---

### Web開発への応用

- **テキストスタイルをデザイントークンで定義する** → `--text-body`, `--text-heading` など
- ライトウェイトフォントは小サイズで使わない（WCAG のコントラスト基準も影響）
- フォントサイズ変更への耐性をレスポンシブ同様に設計する（`rem` 単位の活用）
- アクセシビリティ対応: ユーザーのブラウザフォントサイズ設定を尊重するために `px` ではなく `rem` を使う

---

### OS組み込みフォント一覧

| OS | UIフォント | 等幅フォント | 備考 |
|---|---|---|---|
| macOS / iOS | SF Pro（サンセリフ）、New York（セリフ） | SF Mono | watchOS は SF Compact |
| Windows | Segoe UI | Consolas | Windows 11 から Segoe UI Variable |
| Android | Roboto | Roboto Mono | Noto フォントも同梱 |
| Linux | 環境依存（Ubuntu: Ubuntu Font） | - | デスクトップ環境によってバラバラ |
| 全OS共通 | Arial、Times New Roman | Courier New | デザイン品質は低い（後述） |

**システムフォントスタック（CSS）:**

```css
/* 各OSのベストUIフォントを自動で使う */
font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif;
```

- `-apple-system` → macOS/iOS で SF Pro
- `BlinkMacSystemFont` → Chrome on macOS で SF Pro
- `"Segoe UI"` → Windows
- `Roboto` → Android

### Web開発でよく使うカスタムフォント（Google Fonts等）

**英語UI向け（サンセリフ）:**

| フォント | 特徴 |
|---|---|
| **Inter** | 現代UIの定番。SF Proに近い雰囲気。Notion/Linearなど採用多数 |
| **Geist** | Vercel製。Geist Monoとセットでコード表示にも対応 |
| **DM Sans** | 丸みがあり親しみやすい |
| **Outfit** | クリーンでミニマル |

**日本語対応:**

| フォント | 特徴 |
|---|---|
| **Noto Sans JP** | Googleの多言語統一フォント。安定感あり |
| **BIZ UDGothic** | ユニバーサルデザインフォント |
| **M PLUS 2** | 現代的でUI向き |

**実用的な組み合わせ:**

```css
/* 英語メイン */
font-family: 'Inter', -apple-system, sans-serif;

/* 日本語あり */
font-family: 'Inter', 'Noto Sans JP', -apple-system, sans-serif;
```

> **注: Arial について**
> Arial は1982年にMicrosoftがHelveticaのライセンス料を避けるために作った代替フォント。
> 印刷向け設計でスクリーン最適化がされておらず、字形も古くさい（`a` `g` が一階建て）。
> 現代のUIフォント（Inter / SF Pro / Roboto）と比べると品質が低く、UIには不向き。
> 読めないわけではないが、デザイン品質を気にするなら避けるべき。

OS組み込みフォントで全プラットフォーム統一はほぼ不可能（Arialくらいしか共通しない）。
統一した見た目が必要ならWebフォントを読み込むか、システムフォントスタックで各OSに任せる。

### 補足：ポイント(pt) vs ピクセル(px)

| 単位 | 基準 | 使われる場所 |
|---|---|---|
| **pt** | 論理サイズ（1/72インチ基準） | Apple HIG / iOS・macOS API |
| **px（CSS）** | 論理ピクセル（参照ピクセル） | Web / CSS |
| **dp/sp** | 密度非依存ピクセル | Android |

- Retina (@2x) では `1pt = 2物理ピクセル`、@3x では `1pt = 3物理ピクセル`
- **Web の CSS `px` ≒ Apple の `pt`**（どちらも「見た目の大きさ」基準）

#### CSS px の歴史

- **1990年代**: モニターが低解像度（72〜96dpi）だったので `1px = 1物理ピクセル` で問題なかった
- **CSS 2.1（2010年）**: W3C が**参照ピクセル**を定義 →「96dpi のデバイスを腕の長さから見たときの視角」を基準にした論理単位へ
- **Retina ディスプレイ登場（2010年）**: 物理ピクセルが2倍になったが CSS px はそのまま。ブラウザが自動スケール

> 「どのデバイスでも同じ見た目になること」を優先した結果、px が抽象的な論理単位になった。Apple の pt と同じ発想。

#### プラットフォームごとのフォントサイズの理由

フォントサイズはデバイスの**使用距離（ユースケース）** に基づいて決まっている。論理単位なので解像度・画面サイズは直接関係しない。

| OS | 視聴距離（目安） | デフォルト |
|---|---|---|
| macOS | 50〜70cm（デスク） | 13pt |
| iOS | 30〜40cm（手持ち） | 17pt |
| tvOS | 2〜3m（ソファ） | 29pt |

```
macOS 13pt @ 60cm ≒ iOS 17pt @ 40cm ≒ tvOS 29pt @ 250cm
→ 視角（目に映る見かけの大きさ）としてはほぼ同じ
```

> 「画面が小さいから小さい」「解像度が高いから大きい」ではなく、
> 「そのデバイスを人間がどう使うか（人間工学）」でサイズが設計されている。
> CSS px の参照ピクセル（腕の長さ基準）と同じ発想。

ただし **macOS vs iOS は距離だけでは説明しきれない**。macOS のほうが遠いにもかかわらずデフォルトサイズが小さい理由：

| 要因 | macOS | iOS |
|---|---|---|
| 情報密度 | 高い（多ウィンドウ・プロ向け） | 低い（1タスク集中） |
| 入力手段 | マウス＋キーボード（小さいUIを操作できる） | タッチ（最小タップサイズ 44pt 必要） |
| ユーザー層 | リテラシー高め・集中して使う | 老若男女・片手・流し見も多い |

距離はあくまで一要因。**デバイスのユースケース全体（操作方法・情報密度・ユーザー層）を総合して設計されている。**

## Layout（レイアウト）

### ベストプラクティス

- **関連する項目はグループ化する** → 余白・背景・カラー・区切り線で領域を明確に
- **重要な情報の周囲に十分なスペースを確保する** → 些末な情報を詰め込まない
- **コンテンツは画面/ウィンドウいっぱいに拡張する** → サイドバーやタブバーはコンテンツに重ねて表示

### 視覚的階層

- **コントロールとコンテンツを区別する** → Liquid Glass マテリアルを活用
- **重要な項目は左上に配置する** → ユーザーは上から下・左から右に読む（RTL言語には注意）
- **コンポーネントを位置揃えする** → スキャンしやすくなり階層が伝わる
- **段階的な表示（Progressive Disclosure）** → 画面外にコンテンツがあることを示す（一部を見切れさせる等）
- **コントロール周囲に十分なスペース** → 詰め込みすぎると機能が理解しにくくなる

### 適応性 (Adaptibility, Responsibility)

アプリはデバイス・システムの変化に適応する必要がある。対応すべき変化：

- 画面サイズ・解像度・色空間
- デバイスの向き（縦/横）
- Dynamic Island などのシステム機能
- ダイナミックタイプ（フォントサイズ変化）
- ロケール（RTL言語・日付フォーマット・テキスト長）

**SwiftUI / Auto Layout を使うと動的に適応させやすい。**

### ガイドとセーフエリア

- **レイアウトガイド** → コンテンツ周囲の標準マージン・テキスト幅制限を定義する矩形領域
- **セーフエリア** → ツールバー・タブバー・Dynamic Island などに被らない安全な表示領域

セーフエリアを使うことでデバイス固有の特徴（ノッチ・角丸）との干渉を自動回避できる。

### プラットフォーム別の考慮事項

| OS | 主な注意点 |
|---|---|
| **iOS** | 縦横両方向に対応する。フルサイズ幅ボタンは避ける。ステータスバーは原則表示 |
| **iPadOS** | ウィンドウリサイズに対応。できるだけコンパクトビューへの切り替えを遅らせる |
| **macOS** | コントロールをウィンドウ下部に置かない（ウィンドウが画面下にはみ出すため） |
| **tvOS** | 画面サイズに自動適応しない。セーフエリア上下60pt・左右80ptを守る。フォーカス時の拡大を考慮してパディングを確保 |
| **visionOS** | 重要なコンテンツは中央に。ウィンドウ境界からはみ出さない。ボタンは中心間60pt以上 |
| **watchOS** | ベゼルがパディング代わり。ボタンは横に3〜4つ以上並べない |

### サイズクラス（iOS/iPadOS）

レスポンシブ対応の基本単位。

| クラス | 意味 |
|---|---|
| **Regular Width** | 大きい画面・横向き |
| **Compact Width** | 小さい画面・縦向き |

- iPad はほぼすべての向きで Regular Width / Regular Height
- iPhone は縦向きで Compact Width / Regular Height
- Pro Max / Plus など大型 iPhone は横向きで Regular Width になる

Web 開発での CSS メディアクエリ・Tailwind の `sm:` `md:` ブレークポイントに相当する考え方。

### 補足：フルブリードとステータスバー

**フルブリード（Full Bleed）**
印刷用語で「端まで余白なく塗り潰す」の意。UIではコンテンツが画面端まで余白なく広がるレイアウト。
ゲームや動画など没入感が重要なアプリで推奨。Webではヒーロー画像の `width: 100vw` / `object-fit: cover` が相当。

**ステータスバー（システム）**
- OSのシステムステータスバー（時刻・電池・通信）は**原則常時表示**がベストプラクティス
- 非表示が許容されるのはゲーム・フルスクリーン動画・カメラなど没入感が必要な場合のみ
- 「デザインがすっきりする」という理由だけで隠すのはHIG違反の精神

**アプリ内ステータスバーは別物**
- Slackの「接続中...」バナー、Gmailの送信中インジケーターなどはアプリ独自実装
- システムステータスバーとは無関係で自由に設計できる
- HIGが制約しているのは「OSのステータスバーを勝手に隠すな」という点のみ

### 補足：アダプティブデザイン — ネイティブ vs Web

#### SwiftUI（ネイティブ）

テキストスタイルを指定するだけでダイナミックタイプ・アクセシビリティ・ダークモードが自動で機能する。

```swift
Text("本文").font(.body)       // ダイナミックタイプ自動追従
Text("見出し").font(.headline)

// カスタムフォントでも relativeTo で追従させられる
Text("テキスト").font(.custom("MyFont", size: 17, relativeTo: .body))
```

- `px` 固定指定（`.system(size: 17)`）はダイナミックタイプに追従しないので避ける
- **システムフォント + システム定義スタイルを使えば、アダプティブ対応は特別なことをしなくて良い**

#### Web（Tailwind / CSS）

自分で設計する必要がある。

```css
/* px はユーザーのブラウザフォントサイズ設定を無視する → NG */
font-size: 14px;

/* rem はルートフォントサイズに追従 → OK（Tailwind のデフォルト） */
font-size: 0.875rem; /* = text-sm */
```

- デバイスサイズ対応 → メディアクエリ / Tailwind ブレークポイント（`md:text-base`）
- ユーザーのフォントサイズ設定対応 → `rem` 単位を使う
- 毎回 `sm:text-xs md:text-base` を書くのはアンチパターン → コンポーネント化 or `@apply` で閉じ込める

```css
@layer components {
  .text-body { @apply text-sm md:text-base leading-relaxed; }
}
```

#### まとめ

| | SwiftUI（ネイティブ） | Web |
|---|---|---|
| フォントサイズ追従 | `.font(.body)` だけで自動 | `rem` 単位が必要 |
| デバイス対応 | システムが自動 | メディアクエリ / ブレークポイント |
| ダークモード | 自動 | `dark:` バリアント or CSS変数 |
| コンポーネント化 | `ViewModifier` / カスタムView | `@apply` / Reactコンポーネント |

> HIG を学ぶ意義のひとつは「Apple がネイティブで自動でやっていることを、Web でも意識して再現する」という視点を持てること。

### 補足：フルスクリーン背景の実装

HIG の「コンテンツを画面いっぱいに拡張する」に対応する Web 実装。

#### 基本セット（3つのスタイル）

```css
position: fixed;  /* スクロールしても動かない */
inset: 0;         /* 上下左右の隙間ゼロ = 画面全体を覆う */
z-index: -10;     /* コンテンツの後ろに潜り込む */
```

Tailwind: `fixed inset-0 -z-10`

- `position: absolute` にするとスクロールで一緒に流れる → セクション内背景に使う

#### 実装パターン

| 手法 | 負荷 | 向いている用途 |
|---|---|---|
| CSS アニメーション | 低 | グラデーション・シンプルな動き |
| `<canvas>` 2D | 中 | パーティクル・プロシージャルアート |
| `<video>` | 中（ファイルサイズ大） | 映像・映画的な演出 |
| WebGL / Three.js | 高 | 3Dシーン・インタラクティブ |

```html
<!-- 動画背景の例 -->
<video autoplay muted loop playsinline class="fixed inset-0 w-full h-full object-cover -z-10">
  <source src="bg.mp4" type="video/mp4">
</video>
```

> `muted` がないとブラウザが autoplay をブロックする。

#### object-fit / object-cover

`<img>` `<video>` タグ自身に効くスタイル（子要素には効かない）。背景画像の `background-size` に相当。

```css
object-fit: cover;   /* 縦横比を保ってコンテナを覆う（はみ出した部分はクリップ） */
object-fit: contain; /* 縦横比を保ってコンテナに収める（余白が出る） */
object-fit: fill;    /* コンテナに引き伸ばす（縦横比が崩れる） */
```

フルスクリーン背景に `cover` を使う理由：どんな画面サイズでも隙間なく覆うため。

#### inset プロパティ

`top` / `right` / `bottom` / `left` の一括ショートハンド（CSS 2021年頃〜主要ブラウザ対応）。

```css
inset: 0;              /* 上下左右すべて0 */
inset: 10px 20px;      /* 上下10px, 左右20px */
```

`margin` とは別物。`position: absolute/fixed/sticky` のときの座標指定。

#### pt と px の換算

- `1pt = 1/72インチ`、`1px = 1/96インチ` なので **1pt = 96/72 = 1.333px**
- HIG の数値をそのまま CSS に使うと小さくなる

| Apple pt | CSS px（概算） |
|---|---|
| 11pt（iOS最小） | 約 14.7px |
| 13pt（macOSデフォルト）| 約 17.3px |
| 17pt（iOSデフォルト） | 約 22.7px |
| 29pt（tvOSデフォルト）| 約 38.7px |

## アクセシビリティ (Accessibility)

> あらゆる人がアプリやゲームを楽しめるインクルーシブな体験を設計する

### アクセシブルなUIの3原則

| 原則 | 内容 |
|---|---|
| **直感的** | 慣れ親しんだ一貫した操作で、タスクを明快に実行できる |
| **認識しやすい** | 視覚・聴覚・触覚など複数の方法で情報を伝える |
| **適応性がある** | システムのアクセシビリティ設定に対応し、ユーザの使い方に合わせて変化する |

---

### 視覚 (Visual)

#### テキストサイズ・コントラスト

- **大きなテキストサイズに対応する** — Dynamic Type またはカスタムUIでフォントサイズ拡大に対応（iOS 200%以上、watchOS 140%以上）
- カスタムフォントでも**推奨デフォルト・最小サイズ**を守る（Typography ページ参照）
- 細いフォントウェイトを使う場合は**フォントサイズを大きめに**する

#### カラーコントラスト比（WCAG Level AA 基準）

| テキストサイズ | ウェイト | 最低コントラスト比 |
|---|---|---|
| 17pt 以下 | すべて | 4.5:1 |
| 18pt 以上 | すべて | 3:1 |
| すべて | ボールド | 3:1 |

- システムカラー（`systemRed` 等）を優先 → コントラスト強化モード時に自動でアクセシブルなバリアントに切り替わる
- **カラーだけで情報を伝えない** — 形状・アイコンなどの視覚インジケータも合わせて使う（色盲対応）

#### VoiceOver

- 画面読み上げ機能。全盲・弱視のユーザが画面を見ずに操作できる
- UI要素に適切なアクセシビリティラベルを付ける必要がある

---

### 聴覚 (Hearing)

音声だけで情報を伝えない → テキストベースの代替手段を提供する

| 手段 | 用途 |
|---|---|
| キャプション | ビデオとリアルタイム同期するテキスト |
| 字幕 | テレビ・映画向け台詞のリアルタイム表示 |
| バリアフリー音声ガイド | 映像の休止中に視覚情報を音声ナレーションで補完 |
| 文字起こし | ポッドキャスト・オーディオブック向け完全テキスト化 |

- オーディオキューには**触覚フィードバック**（Haptics）を合わせて提供する
- ゲームなど画面外でイベントが起きる場合は**視覚インジケータ**も追加する

---

### 身体機能 (Motor)

#### コントロールサイズの最低基準

| プラットフォーム | デフォルト | 最小 |
|---|---|---|
| iOS/iPadOS | 44×44 pt | 28×28 pt |
| macOS | 28×28 pt | 20×20 pt |
| tvOS | 66×66 pt | 56×56 pt |
| visionOS | 60×60 pt | 28×28 pt |
| watchOS | 44×44 pt | 28×28 pt |

- 要素間のスペース: ベゼルあり → **12pt**、ベゼルなし → **24pt** のパディング
- 複雑なジェスチャ（マルチフィンガー等）は避け、**シンプルなジェスチャ**を優先
- ジェスチャの**代替ボタン**を必ず用意する（スワイプ削除 → タップして削除ボタンも）
- Siri・ショートカット対応で**声だけでタスク実行**できるようにする
- AssistiveTouch / スイッチコントロール / フルキーボードアクセスに対応する

---

### 認知 (Cognitive)

- UIはシンプル・直感的に。カスタムジェスチャより**システム標準ジェスチャ**を優先
- **時間制限のある要素**（タイマーで自動クローズ等）は極力使わない → 明示的なアクションで閉じる
- オーディオ・ビデオは**自動再生しない**、開始・停止コントロールを分かりやすい位置に配置
- 点滅する光・高速アニメーションに注意 → てんかん・めまいの原因になる
- **「視差効果を減らす」設定**（Reduce Motion）に対応する
  - スプリングアニメーションのバウンスを抑える
  - Z軸の奥行きアニメーションを避ける
  - 移動をフェードに置き換える

---

### Web 開発での実装指針

```css
/* フォーカス可視化（キーボードナビゲーション対応） */
:focus-visible {
  outline: 2px solid currentColor;
  outline-offset: 2px;
}

/* Reduce Motion 対応 */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}

/* コントラスト強化対応 */
@media (prefers-contrast: more) {
  /* より高コントラストなカラーに切り替え */
}
```

- カラーコントラスト確認ツール: [Accessible Perceptual Contrast Algorithm (APCA)](https://www.myndex.com/APCA/)
- `aria-label` / `role` / `alt` 属性でスクリーンリーダー対応
- **カラーだけに頼らない** → アイコン・テキスト・ボーダーで状態を伝える

> ただしデバイスと目の距離も違う（iPhone は手元、Mac は少し遠い）ので単純換算で揃えればいいわけではない。HIGの数値はネイティブアプリの参考値として見る。
