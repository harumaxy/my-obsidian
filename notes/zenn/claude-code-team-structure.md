## 要約

社内 Backlog データ（6プロジェクト、導入前後各6ヶ月）を分析した結果、**固定された少人数の内製チームだけが生産性2.4〜2.5倍**を実現。人の入れ替わりが多いチームはほぼ横ばい〜微減。

### 効果が出る2軸

| 軸 | 内容 |
|---|---|
| ① 継続性 | コアメンバーが固定されている（継続率 vs 効率倍率 r=0.79） |
| ② 一気通貫 | 要件起票者＝実装担当者（一致率91〜100%で高効率、50%で横ばい） |

### フロー分断が Claude Code に特に致命的な理由

- 仕様が凍結され、AI との反復改善ループが機能しない
- 上流の意思決定の文脈が実装者に伝わらず、プロンプト精度が落ちる
- CLAUDE.md などの「文脈資本」が蓄積されない

### 効果が出ない時のチェックリスト

1. コアメンバーを固定する（**最重要**）
2. 実装者が要件・設計も握る（Plan モード活用）
3. CLAUDE.md・ADR で文脈資本を蓄積
4. 少人数で濃く回す
5. AI の出力はレビュー前提で受け取る

> Claude Code は「魔法ではなく**増幅器**」。増幅されるのは、要件分解力・設計力・進め方の型という積み上げた思考フレームワークそのもの。

## 考察・コメント

記事の主張には**交絡変数の問題**がある。

- 人が固定されている → ドメイン知識が蓄積 → 生産性が上がる

これは Claude Code 導入前から成立している関係で、導入後に測っても同じパターンが出るのは自明とも言える。**「固定チームで導入なし」vs「固定チームで導入あり」** の比較がないと、Claude Code 固有の効果かどうかを分離できない。

一点、記事に固有の示唆があるとすれば「一気通貫度（起票＝実装）」の話。分業体制だと仕様の意図がプロンプトに乗らないという問題は、従来開発より AI 開発で影響が大きい可能性はある。

全体としては「良い体制は何をやっても上手くいく」という自明な話を Claude Code の文脈で言い換えた記事。
