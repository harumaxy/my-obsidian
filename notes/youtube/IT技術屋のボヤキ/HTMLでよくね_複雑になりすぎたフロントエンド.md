# HTMLでよくね？複雑になりすぎたフロントエンド

チャンネル: IT技術屋のボヤキ
URL: https://www.youtube.com/watch?v=F7dN7lf83NE

## 一言で言うと

ボタン一つ置くのに大量のツール・設定・ビルド工程が必要になったフロントエンド。しかし2026年、ブラウザ標準機能の進化により「HTMLでよくね？」が現実的な選択肢として戻ってきている。複雑さを「本質的」と「偶発的」に分け、問題に見合った道具を選ぶ力が最も大切。

---

## 第1章：フロントエンドはなぜ複雑になったのか

### 2000年代のシンプルさ
- **3つの道具だけ**：HTML(骨組) → CSS(見た目) → JavaScript(動き)
- **ビルド工程なし**：書いたものがそのままサーバーで動く
- **作成と実行が一致**

### 複雑化のきっかけ
1. **白い待ち時間の問題**：ページ全体の読み込み直しのストレス
2. **AJAX(部分更新)の普及**：ページ全体をリロードせず必要な部分だけ更新
3. **ユーザー体験の向上**：スクロールで自動追加、ボタン即座に反応
4. **JavaScriptの膨張**：画面書き換え + 状態管理が手作業では追いつかない
5. **フレームワークの登場**：React等で複雑な状態管理を自動化

### 「ツールがツールを呼ぶ」時代へ
```
TypeScript(型チェック)→ 変換工程
JSX(特別な書き方) → 変換工程
ファイルまとめる → バンドル
古いブラウザ補う → ポリフィル
...
```
→ **自分のコードとブラウザの間に何層もの変換が積み重なる**

**問題**：1つ1つの道具には理由があるが、全部足し合わせると誰も全体を説明できない重たい仕掛けに

---

## 第2章：複雑さの種類を分ける

### 2つの複雑さ

#### 1. 本質的な複雑さ
- 問題そのものが元々持つ複雑さ
- **避けられない**、どの道具でもシンプルにはならない
- 例：世界中の航空便と価格をリアルタイム管理

#### 2. 偶発的な複雑さ
- 問題ではなく、道具選びのせいで後から付け加わった複雑さ
- **減らせる余地がある**、本来はなくても良かったもの
- 例：ボタン1つ出すのに3つの変換工程

### 現状：2つが混在している
会社の紹介ページ、お知らせ一覧、問い合わせフォーム → **本質的には単純な問題**
なのに→ **複雑なフレームワークを習慣で使ってしまう**

**比喩**：近所の店に毎回大型トラックで行くようなもの
- パンを買うこと < トラックの世話の方が大事になる

## 第3章：ブラウザ進化とHTMLルネサンス

### 標準機能が次々と吸収されている

#### 1. **popover API**
| 昔 | 今 |
|---|---|
| ライブラリで実装 | `popover`属性1つ |
| 重なり順管理 | ブラウザが面倒見る |
| 外側クリック処理 | 自動 |
| ESCキー処理 | 自動 |
| JavaScript | **不要** |

#### 2. **CSS Anchor Positioning**
| 昔 | 今 |
|---|---|
| JavaScriptで位置計算 | CSSで自動計算 |
| ボタン位置に合わせる | 相対配置で自動調整 |
| 画面外対応 | ブラウザが判断 |

#### 3. **View Transitions API**
- ページ遷移をスムーズにアニメーション
- Cross-Document View Transitionsで別ページへの遷移も対応

#### 4. **@starting-style と display: none からのアニメーション**
- `display: none`から表示する際のエントリーアニメーション
- JavaScriptのタイミング制御が不要

#### 5. **フォーム要素のスタイリング**
- checkbox, radio, selectの見た目をCSS + `appearance: none` で自由設計
- わざわざ作り直す必要が消えた

#### 6. **CSS Custom Highlight API**
- DOM要素追加なしで任意のテキスト範囲にスタイル適用
- 検索ハイライト、構文強調に最適

### 2025年以降の現実
- popover：主要ブラウザ全て対応（2025年春）
- Anchor Positioning：Baseline 2026達成（91%のブラウザ対応）
- **ライブラリを入れる理由そのものが減り続けている**

### ライブラリと標準の関係
- 争いではなく、バトンの受け渡し
- ライブラリが切り開いた道を、標準が後からしていく

## 第4章：HTMLで足りる場面、足りない場面

### HTMLで十分な場面 ✅
- **企業紹介ページ**
- **商品一覧**
- **記事を読ませるページ**
- **問い合わせフォーム**

→ 情報を綺麗に見せたいだけ = **本質的複雑さが小さい**
→ 標準機能の手の届く範囲

### HTMLでは辛い場面 ❌
- **スプレッドシート型**：何千ものマス目、変更が一斉に波及
- **複数人同時編集**：状態が刻一刻と動き続ける
- **リアルタイムな状態変化**

→ **本質的に複雑** = 状態管理が頻繁に動く
→ HTMLだけだと処理が散らばる
→ 結局フレームワークを再発明する羽目に

### 結論
- Reactのような複雑な道具が **不要になったわけではない**
- **本当に複雑な問題には今もこれからも強力な道具**
- 大事なのは、複雑さに見合った選択

---

## 第5章：複雑さを身の丈に合わせる

### 「HTMLでよくね？」の本質
- 怠け心ではなく、**問いを立てる健全な態度**
- 「この複雑は本当に必要なのか」と立ち止まる思考

### これからのエンジニアに求められる力

| 昔の価値 | これからの価値 |
|---------|-------------|
| 最新ライブラリをたくさん知る | **問題の複雑さを見抜く力** |
| 複雑な道具を使いこなす | **複雑な道具を使わないと判断する力** |

### 複雑さのマネジメント
- **ゼロにするもの**ではない
- **歴史るもの**でもない
- **身の丈に合わせるもの**

1. 本質的に複雑な問題 → 複雑さを正面から引き受ける
2. シンプルな問題 → 習慣で複雑さを持ち込まない

### 希望：ゆり戻しが起きている
- Webの歴史は複雑さ増加の一方向だった
- **2026年、初めてゆり戻しが起きている**
- ブラウザ進化により、シンプルに戻れる選択肢が増えている

## キーメッセージ

> 複雑な道具を使いこなせることも力。
> でも複雑な道具を使わないと判断できることが、これからはそれ以上に大きな力。

### 次に何かを作る時の思考プロセス
```
手がいつもの重たい道具に伸びる前に

1度だけ心の中で呟いてみる
  ↓
「これ、HTMLでよくね？」
  ↓
✅ そうだ → HTMLで作る
❌ いや、本当に複雑だ → 道具を選ぶ
  ↓
どちらに転んでもあなたはちゃんと考えて選んでいる
それが1番大事
```

## 実践的なTakeway

1. **ブラウザ標準機能の把握**が最優先
   - popover + Anchor Positioning
   - View Transitions
   - フォーム要素のスタイリング
2. **問題の本質を見極める**癖をつける
3. **偶発的な複雑さを減らす**ことに注力
4. **フレームワークは最後の手段**として選択

## 補足：2026年に HTML+CSS だけでできるようになった機能

### 置き換わったライブラリ・パターン

| 機能 | 昔のライブラリ・手法 | 今の標準 |
|------|----------------|---------|
| ポップアップ位置決め | Popper.js, Floating UI | CSS Anchor Positioning |
| ポップオーバー・ツールチップ | Headless UI, Reach UI | popover API |
| ページ遷移アニメーション | Barba.js, custom JS | View Transitions API |
| エントリーアニメーション | setTimeout + JS計時 | @starting-style |
| モーダルダイアログ | Bootstrap Modal | dialog 要素 + トランジション |
| アコーディオン | jQuery Accordion | details/summary |
| フォーム要素スタイル | Custom components | appearance: none + CSS |
| テキストハイライト | Highlight.js等 | CSS Custom Highlight API |
| 自動高さアニメーション | ResizeObserver + JS | interpolate-size |
| スコープ付きCSS | BEM命名規則 | @scope at-rule |
| ネストグリッド | 手動グリッド計算 | CSS subgrid |

### 主要な新機能一覧

- **Popover API**：ESCキー、外側クリック対応、スタッキング自動化（2025年春全対応）
- **CSS Anchor Positioning**：相対位置決めの自動計算（Baseline 2026、91%対応）
- **View Transitions API**：Cross-Document対応、ハードウェアアクセラレーション
- **@starting-style**：display: none からのスムーズトランジション
- **dialog + allow-discrete**：モーダルのアニメーション中もtop-layerに留まる
- **CSS Custom Highlight**：DOM変更なしでテキスト範囲にスタイル適用
- **interpolate-size / calc-size**：height: auto へのアニメーション
- **@scope**：CSS のカプセル化、BEM命名規則不要
- **accent-color**：フォーム要素の色を簡単に変更

### 2026年の現実

✅ **削減できた複雑さ**：ポジショニング、ポップアップ、基本アニメーション、フォームスタイル
❌ **依然必要なもの**：状態管理、複雑なUIロジック、リアルタイム同期

**結論**：シンプルなサイト・ページはもうフレームワークなしで構築可能な時代へ
