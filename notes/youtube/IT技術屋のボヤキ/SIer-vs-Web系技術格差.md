# SIer vs Web系技術格差の実態

YouTube: [IT技術屋のボヤキ - SIer vs Web 系技術格差の実態](https://www.youtube.com/watch?v=Ra5V54odACw)

## SIerのJava開発の現状

### 実際の現場風景
- **Java 8** がまだ現役（2014年リリース、10年以上前）
- **Spring Boot** ではなく独自フレームワーク使用
- **SVN、Eclipse** などのレガシーツール
- **ウォーターフォール開発**プロセス（設計書重視、承認が多い）

### なぜアップグレードできないのか
- 依存ライブラリが対応していない
- テストコストが取れない
- 「動いてるものを変えたくない」という顧客圧力
- セキュリティ対応よりも「変更リスク」を優先する傾向

## 技術格差が生まれる構造的理由

### 1. 顧客の要件に縛られる
- 顧客が「現在の環境を変えたくない」と言えば変えられない
- 新しい技術の承認が下りない
- **技術選定の自由がない** ← これが最大の違い

### 2. 保守契約が長期
- 10年以上動くことを前提
- 銀行、保険、行政システムなど社会インフラ
- リスク回避のため「枯れた技術」を採用
- 「5年後に誰も分からなくなったらどうするか」という不安

### 3. 新技術導入のリスクを取れない
- SIerは契約で納期と品質が決まっている
- 新しい技術で問題が起きたら責任を取らないといけない
- 実績のある技術を使う方が「安全」という判断

## Web系との決定的な違い

| 項目 | SIer | Web系 |
|------|------|-------|
| **サービス形態** | 受託開発 | 自社サービス |
| **技術選定** | 顧客の承認が必要 | 完全に自由 |
| **リリースサイクル** | 長期（ウォーターフォール） | 短期（毎日デプロイ） |
| **開発環境** | SVN、手動デプロイ | Git、CI/CD自動化 |
| **エンジニアの発言権** | 弱い（顧客優先） | 強い |
| **評価される行動** | 安定性重視 | 技術的チャレンジ |

## SIerの価値（単なる「遅れ」ではない）

### 安定性・信頼性が最優先
- 銀行、電力、行政システムが落ちたら社会に大打撃
- 枯れた技術 = 10年間安定稼働が証明されている技術
- 新しい技術も重要だが、使い分けが必要

### 大規模システムのノウハウ
- 何百人・何千人が使うシステムの設計・運用経験
- 数十年のデータを保持するシステム管理
- これは Web系では得られない経験が多い

### 業務知識の深さ
- 金融、保険、物流、行政などの業界知識
- 技術だけでなく業務がわかるエンジニアは少ない
- これは転職市場でも評価される

## 重要な課題と矛盾点

### セキュリティリスク
- **古い = 安全** という論理は実は逆
- Java 8 はサポート終了が近い（またはすでに終了）
- セキュリティパッチが当たらない = **既知の脆弱性を抱えたまま**
- 「動いているものを変えたくない」 > 「セキュリティ対応」という優先順位は危険

### 文化的な問題
- 古臭い謎慣習が多い
- 技術的好奇心や効率性と相反する文化
- これが優秀なエンジニアの流出を招いている

### ビジネスモデルの限界
- 受託開発 = 顧客が主権を持つ
- SIer側が「最適な技術」を提案しても、顧客が承認しなければ使えない
- 本来なら「顧客の技術要件に口出しする資格がない」と言えるが、契約上はそうもいかない

## SIerエンジニアの選択肢

### 1. 現場で学ぶ
- 大規模システム設計
- 長期運用の考え方
- 業務知識（他では得られない）

### 2. 個人で新技術をキャッチアップ
- プライベートで最新技術に触れる
- 環境のせいにせず自分で動ける人になる

### 3. キャリア選択肢
- **Web系への転職** ：技術的チャレンジ、効率性を求める人向け
- **SIer内昇進** ：マネジメント、アーキテクト、PMの道
- 大規模プロジェクト経営経験はSIerでしか詰めない

## 最終メッセージ

> **「どちらが正解ではない。自分が何を大事にするかで決めればいい。」**

大切なのは **どこにいても学び続けること**。

- 環境のせいにして成長を止めるか
- 制約の中でも自分で動けるエンジニアになるか

これがどの環境でも活躍できるエンジニアになるための鍵。

## 個人的な所感

SIerは技術的好奇心や効率性を求めるエンジニアには向いていない。しかし、セキュリティ対応（古いバージョンの放置）という根本的な課題が解決されない限り、「安定性」という名目も説得力がない。本来は「安定 + 最新セキュリティパッチ」を両立させるべき。
