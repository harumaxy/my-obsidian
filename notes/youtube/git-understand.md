# Gitを本当に理解していますか

【出典】IT技術屋のボヤキ - YouTube  
【URL】https://www.youtube.com/watch?v=vCP73gPBwT8

## 概要

毎日Gitを使っているけど、仕組みはよくわかってない…という人向けの解説動画。多くのエンジニアが勘違いしている4つのポイントを理解するだけで、Gitへの恐怖感が消える。

## 多くのエンジニアが勘違いしている4つのポイント

### 1. GitとGitHubは別物

**関係性：動画とYouTubeの関係に近い**

| | Git | GitHub |
|---|---|---|
| 役割 | バージョン管理ツール（コンテンツそのもの） | Gitで管理したものを置く場所（プラットフォーム） |
| 依存性 | 単独で動作可能 | GitHubがなくても使える |
| その他 | 2005年にLinus Torvaldsが作成 | 2008年にサービス開始（Gitより3年後） |

**重要：** GitはローカルだけでOK。インターネットがなくてもコミット、ブランチ、リモートなしでも完全に動作する。GitとGitHubを混同するとGitの本質が見えなくなる。

---

### 2. コミットは「差分」ではなく「スナップショット」

**イメージ：写真のようなもの**

- 前回からの変更点を保存しているのではなく、その瞬間のプロジェクト全体を丸ごと保存
- 過去のコミットに戻りたいときに、差分を逆算して適用する必要がない

**メリット：高速アクセス**

- 1万5000コミット前の状態を見たい場合、差分方式だと変更を1万回適用する必要がある
- スナップショット方式なら「タイムマシンで直接その時代に行く」ようなもの
- `git checkout` が一瞬で終わるのはこの設計のおかげ

**実装上の工夫：**
- 概念としては全体のスナップショットだが、実装上は効率的に圧縮
- 変更のなかったファイルは前のコミットへの参照のみ持つため、重複しない

---

### 3. ブランチは「コピー」ではなく「ポインタ」

**イメージ：本のしおり**

- ブランチを切ってもファイルは**1バイトもコピーされない**
- ただポインタ（しおり）が増えるだけ
- 本自体は1つで、しおりが複数あるイメージ

**メリット：軽量で高速**

- 1GBのプロジェクトでもブランチ作成は0.1秒で終わる
- 昔のSVNはブランチを切ると本当にファイルをコピーしていたため、数分かかることもあった
- Gitは気軽にブランチが切れるからこそ、フィーチャーブランチワークフローが流行った

**ブランチ切り替え時：**
- HEAD（「自分がいる場所」を指すしおり）が別のブランチに移動
- ワーキングディレクトリのファイルはそのコミットの状態に更新される
- 内部的には「しおりが動いただけ」

---

### 4. マージとリベースは「履歴が違う」

**マージ：川の合流**

```
    feature
   /
main ----○---- (merge commit)
```

- 2つの川が合流するポイントを作る
- 「ここで合流しました」という事実が履歴に残る
- 履歴が正直（どのタイミングでマージしたか明確）

**リベース：川の源流を付け替え**

```
feature ----○
            |
main   -----+----○ (rebase した後)
```

- 川の源流を付け替える
- 「最初からこっちの川から流れ出ました」みたいに歴史を書き直す
- コミットのハッシュが変わる

**使い分け：**

| | 正直さ | 見た目 | 危険性 |
|---|---|---|---|
| **マージ** | ✅ 履歴が正直 | ごちゃごちゃ | プッシュ後でも安全 |
| **リベース** | 歴史を改ざん | 一直線で綺麗 | ⚠️ プッシュ後は危険 |

**リベースが危険な理由：**
- 他の人が古い歴史を見ているのに、自分だけ歴史を書き換える
- 「あれ？俺の知ってる歴史と違う」となって混乱する

**よくあるチームルール：**
> ローカルで作業中はリベース。共有したらマージ。
> 自分だけの歴史は書き換えてOK。みんなの歴史は書き換えない。

## Gitを理解することで得られる効果

### 1. トラブルに強くなる

- 「なんか変なことになった」という時に、何が起きてるか理解できる
- 今までは「ググってコピペしたコマンド」をよくわからないまま実行していた
- 仕組みが分かれば「このコマンドは履歴を書き換える」「このコマンドはポインターを動かすだけ」と判断できる

### 2. コマンドの違いが理解できる

**例1：`git reset` のオプション**
- `--soft` ：しおり（HEAD）だけ動かす
- `--hard` ：しおり（HEAD）を動かして、作業中のファイルも巻き戻す

**例2：その他のコマンド**
- `git fetch` vs `git pull` の違い
- ポインタの概念が分かればオプションの意味も見えてくる

### 3. 恐怖感がなくなる

- Gitが怖いのは「中身がブラックボックス」だから
- 中身が分かれば怖くない
- 毎日使うツールだからこそ、表面的に使うのと仕組みを理解して使うのでは全く違う

## コアメッセージ

**この4つを押さえるだけでGitの理解が一気に深まる：**

1. **GitとGitHubは別物** → 分散バージョン管理の本質を理解
2. **コミットはスナップショット** → 高速アクセスが可能
3. **ブランチはポインタ** → 軽量で気軽に切れる理由
4. **マージとリベースは履歴が違う** → 使い分けで履歴管理が変わる

**実践のコツ：**
> 「このコマンドは何をしてるんだろう？」と考えながら使うと、どんどん理解が深まる
