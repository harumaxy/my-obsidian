## Java の歴史と成功

### 誕生と革新

- **1995年**: Sun Microsystemsが発表
- **キャッチフレーズ**: 「Write Once, Run Anywhere」
- **革命的な特徴**: JVM（Java Virtual Machine）上で動作 → プラットフォーム非依存
  - Windows, Mac, Linux など異なるOS上で同じコードが実行可能
  - 当時は革新的

### 覇権を握れた理由

1. **プラットフォーム非依存性（JVM）**
   - OSを選ばない → 企業にとってコスト削減につながる
   - サーバー変更時、コード書き直し不要

2. **ガベージコレクション（自動メモリ管理）**
   - C/C++：手動でメモリ確保・解放 → バッファオーバーフロー、メモリリークのリスク
   - Java：自動 → 開発効率向上、安全性向上

3. **大企業による採用とエコシステムの充実**
   - J2EEで大規模業務システムの標準に
   - 銀行、保険、企業システム全てがJavaベース
   - Struts, Spring, Hibernateなど優秀なフレームワークが登場

4. **人材確保と教育体系**
   - 2000年代は求人数トップクラス
   - 「Javaを学べば食いっぱぐれない」という評判

## 衰退の転換点：2009年

### Oracleによる買収と影響

- **2009年**: OracleがSun Microsystemsを買収
- **文化の変化**: Sunのオープン文化 → Oracleの商業的姿勢
- **象徴的な出来事**: Google vs Oracle 訴訟
  - AndroidがJavaのAPIを使用していることに対する著作権訴訟
  - 10年以上の泥沼の裁判に発展
  - 開発者に「Javaを使うと訴えられるかもしれない」という不安を与えた

### ライセンス体系の複雑化

- Oracle JDK：商用ライセンス料が必要な時期あり
- ユーザーからの信頼喪失
- OpenJDK などの代替実装が台頭（Amazon Corretto, Azul Zulu, Eclipse Adoptium）

### 他言語の台頭

- **Python**: データサイエンス、機械学習で爆発的成長
- **Go**: クラウドネイティブな軽量言語として登場
- **Kotlin**: 2019年、GoogleはAndroidの公式推奨言語に変更
  - JavaからKotlinへの移行 = Javaの地盤喪失

## 衰退の本質的な理由

### 1. 冗長な構文（ボイラープレート）

- 簡単な処理にも大量のコード必要
- ゲッター、セッター、コンストラクター
- 他の言語なら数行で済む → Javaなら数十行

### 2. 進化の遅さ

- 後方互換性を重視 → 新機能追加が遅い
- **ラムダ式**: 他の言語では標準 → Javaは2014年にようやく導入
- 関数型パラダイム追従に大幅遅延
  - OCamlやLispなど1970年代の言語でもラムダ計算機能がある

### 3. クラウドネイティブ時代への適応遅れ

- **JVM の欠点**:
  - 起動時間が遅い（秒単位）
  - メモリ消費が多い
- **要求される環境**:
  - コンテナ、マイクロサービス、サーバーレス
  - 軽量で起動が速い言語が必須
- Go, Rust, TypeScript がこの需要に応える

### 4. 若い開発者の離脱

- 「古い」「ダサい」「書くのが面倒」というイメージの定着
- 新規採用でJavaが選ばれなくなる
- 教育現場でもJava軽視傾向

### 5. Oracleのイメージ悪化

- 訴訟好きで有名
- クローズドな商業的姿勢
- OSS界での「悪の帝国」というイメージ
- 業界全体がOSS・自由の価値を重視している時代との乖離

## 現在と今後

### Javaは完全には死んでいない

- **TIOBE Index**, **RedMonk** でも依然上位
- **エンタープライズ市場**: 大規模業務システムはまだJavaベース
- **既存システム保守**: 大きな需要がある

### 復権の兆し

- **Spring Boot**: Web開発の敷居を大幅に下げた
- **最近の改善**: 半年ごとに新バージョンリリース
- **新機能の追加**:
  - レコード型
  - パターンマッチング
  - 他言語の良さを積極的に採用

### ただし課題は残る

- **新規プロジェクト**: スタートアップはPython, TypeScript を選ぶ傾向
- **言語選定の分散化**:
  - GraalVM（Oracle、ライセンス不安）
  - Quarkus（RedHat、知名度低い）
  - Micronaut（独立系、マイナー）
  - Kotlin/Scala（Better Javaだが、JVM起動時間問題は引き継ぐ）
- **イメージの固定化**: 「レガシーシステムの言語」という立場

### Android はJavaの最後の砦

- **30億デバイス** 以上がAndroid
- **ART (Android Runtime)**: JVMの改進版、モバイル最適化
  - Dalvik VM から AOT + JIT ハイブリッドに進化
  - 起動速度、電池持ちが大幅改善
- **dex形式**: Javaバイトコードをモバイル用に最適化
- **マルチメーカー環境**（Samsung, Xiaomi等）での統一開発環境として必須
  - AOSP をベースに各メーカーがカスタマイズ
  - アプリはプラットフォーム非依存で動作
- Kotlinも推奨されているが、基盤はJava/Kotlin
- これがあるため、Javaエンジニアの需要はゼロにならない

## Javaエンジニアのキャリア選択肢

### パターン1: Javaを続ける

- エンタープライズ市場は今後も存在
- Spring Boot, クラウド対応スキルを磨く
- 需要は確実にある（高年収も期待できる）

### パターン2: 他言語も学ぶ

- Kotlin は Java知識があれば学習容易
- サーバーサイドキャリアを広げたい場合は有効
- 1つの言語に依存しない方がキャリア選択肢は広がる

### 重要な視点

> Javaほどの言語でも、時代の流れで立場が変わる。
> 今学んでいる言語が10年後も主流とは限らない。
> 常にアンテナを張って、変化に対応できる準備をしておくことが、
> 長く活躍できるエンジニアの姿勢。

## 関連考察

### ランタイムの進化と戦略の違い

#### Dalvik VM → ART への進化

- **Dalvik（2005-2014）**: JIT 戦略
  - 実行時にホットパスをネイティブコード化
  - 起動が遅い、電池消費が多い
  
- **ART（2014-）**: AOT + JIT ハイブリッド戦略
  - インストール時に dex をマシンコード変換
  - 起動、実行速度が大幅改善（3-5倍）
  - 電池消費削減
  - インストール時間が長くなるというトレードオフ

#### 複数ランタイム実装の存在

- **HotSpot JVM**: サーバー向け（JIT戦略、実行時最適化重視）
- **GraalVM Native Image**: サーバーレス向け（AOT戦略、起動速度重視）
- **ART**: モバイル向け（AOT+JIT ハイブリッド、電池・起動時間重視）

それぞれが異なるユースケースに最適化。ただし選択肢の増加は、初心者や組織にとって判断コストになり、「Javaから離れる」きっかけになっている側面もある。

### Androidエコシステムの構造

マルチメーカー環境（Samsung, Xiaomi, Oppo等）では、**プラットフォーム非依存のJVM**が必須。

- C/C++ では各ベンダーごとにハードウェア最適化が必要 → 開発効率低下
- Java は同じコードがすべてのデバイスで動作 → 開発効率向上

**AOSP（Android Open Source Project）**
- Googleが公開（Apache 2.0）
- 各メーカーがカスタマイズ（UI、プリインストールアプリ等）
- アプリはART対応dex形式なら全デバイスで動作

この構造が Javaの「Write Once, Run Anywhere」の理想を、唯一の大規模プラットフォームで体現している。

### OS によるプロセス管理の厳しさ

Android では OS が**メモリ管理と電池節約を最優先**：

- **Low Memory Killer**: メモリ不足時に OS がバックグラウンドアプリを強制終了
- **Doze モード**: 数時間 idle 状態だとバックグラウンド処理を制限
- **App Standby**: 使用していないアプリを活動制限
- アプリ側からは制御不可

デスクトップ OS（Linux, Windows, macOS）では、プロセスは起動後明示的に終了まで動くが、Android では OS に主導権を握られている。これが開発者体験と開発難易度に大きく影響。
