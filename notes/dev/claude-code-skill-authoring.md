## なぜ Skill を使うのか(CLAUDE.md / rules との使い分け)

| 手段 | ロード | 向いているケース |
|------|--------|------------------|
| `CLAUDE.md` | 毎セッション常時 | プロジェクト全体の憲法・命名規約 |
| `.claude/rules/*.md` | ファイルパス一致時 | 特定拡張子を触るときだけ効かせたい規約 |
| **Skill** | 依頼に応じてオンデマンド | ○○を使った実装依頼のときだけ発火させたい |

**Skill の利点**: 起動時は name と description のみロード。本体は必要時だけ読まれるため、いくつ作ってもコンテキスト負荷がほぼゼロ。

## Skill の配置場所

```
~/.claude/skills/          # 個人用・全プロジェクト横断
.claude/skills/            # プロジェクト用・git でチーム共有(同名は project 優先)
```

## ディレクトリ構成(progressive disclosure)

```
my-lib-skill/
├── SKILL.md          # 目次役。500行未満を死守
├── reference/
│   ├── api.md        # 主要 API チートシート
│   ├── idioms.md     # ライブラリ特有の作法・イディオム
│   └── gotchas.md    # ハマりどころ・アンチパターン
└── examples.md       # 入出力の具体例
```

- `SKILL.md` は目次として機能させ、詳細は別ファイルへ誘導
- 参照は SKILL.md から **一段だけ** 深く張る(ネスト禁止)
- 深くネストすると Claude が部分読みで情報を欠かす

## description の書き方(最重要)

Skill が発火するかどうかは **description で決まる**。

```yaml
---
name: implementing-with-htmx
description: |
  HTMX を使った実装のベストプラクティス、属性の正しい使い方、よくあるアンチパターンの回避。
  HTMX、hx-* 属性、hypermedia 駆動の UI を実装・レビューするときに使用する。
---
```

- 「what」(何をするか)と「when」(いつ使うか)の両方を書く
- 「this skill does X」ではなく「use this skill when you need to X」
- 具体的なトリガー語: ライブラリ名・関連キーワード・拡張子・典型的な依頼文言
- name は動名詞形、英小文字・ハイフンのみ、64文字以内

## 蒸留のルール

### 入れるべきもの
- **Claude が間違えやすい箇所**(既知情報は不要)
- 推奨 API の正しい使い方
- 非推奨・レガシーパターンとの違い
- ライブラリ特有のイディオム
- よくあるハマりどころ・アンチパターン

### 入れてはいけないもの
- 「○○とは何か」的な一般説明(Claude は既に知っている)
- バージョン依存で陳腐化する記述を地の文に直書き
- 時限的な情報は「old patterns」セクションに隔離

**重要**: URL を列挙するだけでは効果が薄い。正典を Claude に蒸留させてローカル Markdown として同梱する。

## 正典ソースの選び方(C ライブラリ系)

| ライブラリ | 正典の在り処 | 注意 |
|-----------|-------------|------|
| **sokol** | `sokol_*.h` のヘッダコメント + `sokol-samples/` | 別立て Docs なし。ヘッダが唯一の正典 |
| **SDL3** | wiki.libsdl.org + README-migration(SDL2→3 移行ガイド) | 世の SDL 情報の大半は SDL2。差分集約の移行ガイドが最重要 |
| **raylib** | raylib.com/cheatsheet + `raylib.h` + 公式 examples | examples でゲームループのイディオムを蒸留 |
| **zig** | 固定バージョンの公式 build-system ガイド + 実例の build.zig | 変化が速すぎる。**第三者ブログは使わない** |

**ブログ・記事の使い道**: 設計思想の理解と破壊的変更の経緯把握のみ。API の細部には使わない。

**URL より「ローカル clone のファイルパス」を渡す**方が確実(sokol などは特に)。

## Skill 生成ツール

### skill-creator (公式メタスキル)
- インタビュー → `allowed-tools` 選定 → `SKILL.md` 生成 → テスト → A/B 評価
- **蒸留は自分で行う**。skill-creator は構造化・トリガー設計・評価ループが役割
- 評価ループ: 「skill あり vs なし」の合格率・トークン・時間を数値比較

### Firecrawl skill-gen (第三者)
- ドキュメント URL を渡すと frontmatter・ガイドライン・アンチパターン付き SKILL.md を生成
- **単一ページ前提** → sokol(ヘッダ)や zig(マルチページ+版依存)には不向き
- SDL3/raylib の初稿生成には有効。生成後の精製が必須

### 使用前の注意
第三者 skill は frontmatter が Claude のシステムプロンプトに入るため、**中身を必ず確認してから有効化**すること。

## 現実的な構築フロー

1. **正典をローカルに clone** する(sokol, SDL3, 各 zig バインディングのリポジトリ)
2. **skill-creator を土台に起動**し、「この正典ファイル群を蒸留して `reference/xxx.md` を作って」とファイルパスで指示
3. **自動では入らない部分を手で足す**: 判断・アンチパターン・対象 zig バージョンの明記
4. **評価ループを回す**: 「skill あり/なし」で出力が本当に良くなっているか数値確認
5. **セッション中の編集は即反映**(ただし skills ディレクトリ自体を新規作成した場合のみ再起動要)

## 自動化できること vs 人が握ること

| 自動化できる | 人が握る |
|-------------|----------|
| 構造のスキャフォルド | 「何を残すか」の判断 |
| 初稿の蒸留 | アンチパターンの追記 |
| テストケース実行 | バージョン固定・明記 |
| A/B 評価の計測 | 「本当に効いているか」の判断 |
