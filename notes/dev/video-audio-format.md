# 動画・音声フォーマット まとめ

## コーデックとコンテナの違い

- **コーデック**: 映像・音声を圧縮/展開するアルゴリズム（H.264、AAC など）
- **コンテナ**: コーデックを「入れる箱」。映像・音声・字幕を束ねるファイル形式（MP4、MKV など）

## コーデックが何を記録しているか

- **空間圧縮（イントラ）**: 1枚の画像内で JPEG のような圧縮
- **時間圧縮（インター）**: 前後フレームとの差分だけ記録。静止画が多いほどサイズが小さくなる
- **ピクセル情報**: RGB ではなく YCbCr（輝度+色差）で記録。人間の目は色より明るさに敏感なので色を間引く

AV1 はこれらのアルゴリズムが H.264 より大幅に洗練されていて、同じ画質でファイルが小さい。代償としてエンコードに時間がかかる。

## 動画コーデック

| コーデック | 別名 | ブラウザ/OS互換 | 圧縮効率 | CPU負荷 | 一言 |
|---|---|---|---|---|---|
| **H.264** | AVC | ほぼ全て | 普通 | 低（HW対応広い） | **定番。これで困らない** |
| **H.265** | HEVC | Safari◎ Chrome△ | H.264の2倍 | 中（特許問題あり） | Apple系で強い |
| **VP9** | - | Chrome/Firefox◎ Safari△ | H.265並 | 中 | YouTube が使ってる (Safari の互換性がないが、 H.264 も併用で配信している) |
| **AV1** | - | 最新ブラウザ全対応 | VP9の30%増 | 高（エンコード） | **次世代の定番候補** |
| **H.266** | VVC | ほぼ未対応 | AV1の2倍 | 超高 | エッジ・まだ普及前 |

## 音声コーデック

| コーデック | 互換性 | 特徴 | 用途 |
|---|---|---|---|
| **AAC** | ほぼ全て | 良品質・小サイズ | **動画埋め込みの定番** |
| **MP3** | 全て | 古いが枯れてる | 音楽配信レガシー |
| **Opus** | Chrome/Firefox◎ Safari◎(iOS11+) | 低遅延・高品質 | **リアルタイム通話/配信の定番** |
| **FLAC** | ブラウザ対応△ | 可逆圧縮 | 音楽アーカイブ |
| **AC-3/E-AC-3** | Netflix等 | Dolby Audio | 映像配信サービス |

## コンテナ（ファイルフォーマット）

| コンテナ | 拡張子 | よく入れるコーデック | 用途 |
|---|---|---|---|
| **MP4** | .mp4 | H.264 + AAC | **汎用の定番** |
| **WebM** | .webm | VP9/AV1 + Opus | Web 配信 |
| **MKV** | .mkv | 何でも | ローカル保存・アーカイブ |
| **MOV** | .mov | H.264/H.265 + AAC | Apple 系 |

## 「これを使っておけば」まとめ

```
Web配信・汎用:     H.264 + AAC in MP4
YouTube/Web配信:   VP9 or AV1 + Opus in WebM
次世代Web:         AV1 + Opus in WebM（Safari対応が整ってきた）
Apple向け:         H.265 + AAC in MOV/MP4
音声通話/配信:     Opus
高音質アーカイブ:  FLAC
```

## コンテナとコーデックの組み合わせ制限

コンテナによって入れられるコーデックに制限がある。

**厳しい制限（仕様上入れられない）**

| コンテナ | 制限 |
|---|---|
| **WebM** | VP8/VP9/AV1 + Vorbis/Opus 専用。H.264・AAC は入れられない |
| **MP4** | Theora・Vorbis などの OSS コーデックは非対応 |
| **MKV** | 制限なし。何でも入れられる |

制限がある理由は、コンテナがトラックのメタデータをどう記述するか仕様が決まっているため。WebM は Google が AV1/VP9 専用に設計したので最初から H.264 を入れる仕様がない。

**MKV の例（映画リップでよくある構成）**

```
video:           H.264
audio track 1:   日本語 AAC
audio track 2:   英語 AAC
subtitle track 1: SRT（テキスト字幕）
subtitle track 2: PGS（画像字幕）
```

## 命名の由来

| 名前 | 意味 |
|---|---|
| **H.264/265/266** | ITU-T の「H シリーズ（視聴覚・マルチメディア系）」勧告の連番。264・265・266 に深い意味はない |
| **VP9** | Video Processor の略（Google）。VP3 → VP6 → VP8 → VP9 と続いた連番 |
| **AV1** | AOMedia Video 1 の略。AOM（Alliance for Open Media）が作った最初のコーデック |

---

## 2つの陣営：標準化団体 vs オープン

```
標準化団体（ITU-T/ISO）陣営        オープン陣営（ロイヤリティフリー）
H.264 → H.265 → H.266             VP9 → AV1 → (AV2 開発中)
ライセンス料あり                    ロイヤリティフリー
Apple・放送・映像業界が強い          Google・Netflix・ブラウザ系が強い
```

H.265 は特許プールが複数あり、ライセンス料が複雑で Google・Mozilla が採用を拒否。これが AV1 誕生のきっかけ。

AV1 は Google 単独だった VP9 と違い、Google・Mozilla・Microsoft・Amazon・Netflix・Intel・ARM 等が **Alliance for Open Media（AOM）** を結成して共同開発（2018年公開）。特許を持ち寄って相互免除する形でロイヤリティフリーを実現している。

Safari が VP9/AV1 に消極的だったのも、Apple が H.265 に投資していた背景から。最近は AV1 にも対応してきた。

## 開発者として何を意識するか

**基本は気にしなくていい**（ライブラリ・ブラウザが処理してくれる）。

知っておくと役立つ場面：

- ブラウザで動画が再生できない → コーデックの互換性問題の可能性
- `<video>` で複数の `<source>` を用意するとき（WebM と MP4 を両方置くなど）
- WebRTC でリアルタイム通話を実装する → Opus/VP8/VP9 の選択が絡む
- モバイルでバッテリー消費が問題 → ハードウェアデコード対応コーデックを選ぶ話になる

YouTube のような大規模サービスは複数コーデックでエンコードしてブラウザごとに出し分けている（Chrome → VP9/AV1、Safari → H.264/H.265）。

## 音声コーデックの使い分け詳細

| コーデック | ユースケース |
|---|---|
| **AAC** | 動画配信・音楽配信の汎用。`<video>` タグや MP4 に入れる標準 |
| **Opus** | リアルタイム通話・チャット・文字起こしサービス。遅延最小（2.5ms〜）。WebRTC の標準 |
| **FLAC** | 音楽アーカイブ・マスタリング。完全可逆。再生配信用途ではなく保存用 |
| **AC-3 / E-AC-3** | 5.1ch/7.1chサラウンド。Blu-ray・Netflix・放送業界向け。Dolby ライセンス必須。開発者が選ぶものではなくコンテンツに付いてくる |

**Web 開発の使い分け指針**

- 動画・音楽配信 → **AAC**（MP4 に入れて `<video>` タグで再生）
- チャット・録音・文字起こし → **Opus**（Whisper 等の音声認識モデルへの入力も問題なし）

ブラウザの `MediaRecorder` API はデフォルトで Opus（WebM コンテナ）を出力するので、録音系は何も指定しなくてもそうなる。

## 音声コーデックの圧縮手法

動画の「フレーム差分」とは仕組みが異なる。

### 心理音響モデル（AAC・MP3・Opus・AC-3）

人間の耳の特性を利用した「知覚符号化」：

- **周波数マスキング**: 大きな音の隣の周波数は聞こえない → 捨てる
- **時間的マスキング**: 大きな音の直前・直後の小さな音は聞こえない → 捨てる
- **MDCT（変形離散コサイン変換）**: 時間軸の波形を周波数軸に変換してから量子化（JPEG の DCT と同じ発想）

### 線形予測（FLAC）

動画の「前フレームとの差分」に近い仕組み：

```
次のサンプル値 ≒ 前のサンプルの線形結合で予測
→ 予測との誤差（残差）だけ記録
→ 残差はエントロピー符号化（Rice 符号）で圧縮
```

### ステレオ符号化

左右チャンネルを直接保存せず **和（M）と差（S）** に変換して記録。差は通常ほぼゼロに近いので小さく圧縮できる。

### 動画との違い

音声の波形は常に変化するため「フレーム単位の差分」ではなく、**周波数ドメインでの量子化 + 心理音響モデルによる間引き**が主役。
