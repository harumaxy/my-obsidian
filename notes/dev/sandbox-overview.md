# Sandbox 系サービスの流行とは

Cloudflare Sandbox / Vercel Sandbox / E2B / Modal / Daytona など、**AI のための実行環境(サンドボックス)** サービスが流行している背景と、従来の Docker コンテナホスティングとの違いをまとめる。

## なぜ今サンドボックスが流行っているのか

### 1. LLM・AIエージェントの実用化

直接の引き金は LLM が生成したコードを安全に走らせる必要が出てきたこと。AIエージェント、データ分析ツール、UIフレームワークなどが、**信頼できない動的生成コード**を実行しなければならなくなった。

### 2. 脅威モデルの変化(プロンプトインジェクション)

単に「他人のコードを動かす」だけでなく、**prompt injection** という固有のリスクが加わった。

- 生成コードは入力によって形作られ、その入力は攻撃者に操作されうる
- 生成コードがアプリ本体と同じ実行環境を共有すると、認証情報・ネットワーク・ファイルシステムをそのまま継承してしまう
- = 「LLM にターミナルへのフルアクセスを渡している」状態になりかねない
- → 隔離された使い捨て環境に逃がす必然性が生まれた

### 3. エージェント特有のワークロード形状

エージェントの振る舞いは「ファイル確認 → 計算実行 → API 呼び出し」といった**素早く反復的なツール呼び出し**を伴う。

- それぞれが独立したサンドボックスを必要とする
- 結果として**数千ものサンドボックスを一時環境として高速に生成・削除**し続ける必要がある
- 起動時間がボトルネックになり、スピンアップが遅いとエージェント全体が停滞してインタラクティブ性が死ぬ

## Docker コンテナホスティングとの違い

違いは大きく **「セキュリティ境界」** と **「運用・ライフサイクルモデル」** の2軸。

### 軸1: 隔離境界 — 共有カーネル vs 専用カーネル

これが最も本質的な違い。

| | 従来コンテナ (Docker) | microVM (Firecracker など) |
| --- | --- | --- |
| カーネル | ホストカーネルを全コンテナで**共有** | **独自カーネル**(ゲストカーネル)を持つ |
| 隔離 | カーネル脆弱性/設定ミスでコンテナエスケープ → ホストや他コンテナに波及 | KVM 経由でハードウェアレベル隔離。ゲスト内部が侵害されてもホストや他VMに直接波及しない |
| 起動 | 速い | Firecracker で約 125ms |
| 脅威モデル適合 | プロンプトインジェクションを含むLLM生成コードには**不適合** | 適合 |

- runc・containerd・カーネルの CVE が 2023〜2026 で繰り返し出ているのは現実の問題
- 中間的な選択肢として **gVisor**(Modal が採用、syscall をユーザー空間でインターセプト)もある
- Docker 自身も「ワークロードの信頼プロファイルに隔離を合わせる」のが正解と認めている。アーキテクチャはどこも **「内側はコンテナ、境界が必要な外側はVM」** に収束

### 軸2: ライフサイクルと開発者体験 — 命令的な「プリミティブ」

従来のコンテナホスティング (ECS / Cloud Run / 生 Kubernetes) は「**長く生きるサービスをデプロイする**」モデル前提。エージェントが求める「アプリのロジックから直接、短命の環境を大量に生成・破棄する」用途には素直に乗らない。

> Kubernetes 自身もこのギャップを認め、Agent Sandbox プロジェクトで「安定したID・ライフサイクル制御(pause/resume/予定削除)・強い隔離」を埋めようとしている。

新世代サンドボックスは **SDK ファーストの「プリミティブ」** として抽象化:

- **プログラマブルな生成・破棄**: 名前で呼ぶとオンデマンド起動、アイドルで自動スリープ、新リクエストで復帰。同じIDでどこからでもアクセス、コマンド実行・リポジトリclone・ファイル書込・プロセス管理を SDK で操作
- **コールドスタート最適化**: 起動速度が競争軸(Blaxel 25ms、E2B/Daytona 約90〜150ms)。WarmPool で事前ウォーム
- **状態管理 (snapshot / fork / hibernation)**: 同じ状態から VM を20回 fork して20通りの経路を試す、といったエージェント特有のニーズに対応
- **ネットワーク層でのシークレット注入**: Cloudflare の例ではアウトバウンドプロキシが外向きリクエストを横取りし、ネットワーク層で認証情報を注入。エージェント自身はトークンを一切見ない

## 各サービスの下回り(隔離技術)はバラバラ

「microVM = 理想形」だが、全サービスが Firecracker を使っているわけではない。**信頼プロファイルに応じた強い隔離レイヤー**を選んで使っている。

| サービス | 隔離技術 | 主SDK |
| --- | --- | --- |
| E2B | Firecracker microVM | TS / Python 両対応 |
| Vercel Sandbox | Firecracker microVM | TS 寄り |
| Cloudflare Sandbox | **Firecracker ではなく** Containers + Durable Objects ベース | TS 寄り |
| Modal | コンテナ + gVisor | Python 寄り |
| Daytona | 標準 Docker コンテナ(オプションで Kata) | - |
| Beam | - | Python 寄り |

- エージェントのオーケストレーション側が Python (LangChain / CrewAI) で書かれることが多く、Python も一級市民

## 「サーバーレスから呼ばれる」は半分正しい

- サンドボックスを呼ぶ**制御側(エージェント/アプリのロジック)**は、Workers / Vercel Function に限らずどこで動いてもよい。普通のバックエンドサーバーからでも OK
- Cloudflare / Vercel が自社サーバーレスとの統合を強調するのは、技術的必須要件ではなく**エコシステムの引力**(認証トークン自動連携・課金ダッシュボード統合・運用一元化)が売りだから
- E2B のようにプラットフォーム非依存で「どこからでも SDK で叩ける」中立型もある

## まとめ

> コンテナの共有カーネルは LLM 生成コード(+プロンプトインジェクション)の脅威モデルに対して弱い。
> そこで **microVM や gVisor 等の強い隔離**を土台にしつつ、生成・fork・起動・停止・再開・シークレット注入などを **SDK でプログラマブル**にし、warm pool や snapshot でエージェント特有の **「大量・短命・低レイテンシ」** 要求に応えたのが現代の Sandbox 系サービス。

これは「Docker が時代遅れになった」話ではなく、**脅威モデルが変わったことへの業界全体の応答**。判断基準の定番は「もしこのサンドボックスが侵害されたら最悪何が起きるか」:

- 影響が現在のタスクに限定される → **コンテナで十分**
- 同一ホスト上の他ユーザーのデータに及びうる / 公開され攻撃者の動機が強い → **microVM が必要**

Docker 自身も「信頼できるワークロードを自分で管理しているなら Docker で十分」「外部ユーザーや AI エージェントなど制御できないコードを動かすなら Firecracker が狙って排除しているリスクが入り込む」と認めている。
