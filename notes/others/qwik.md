# Qwik - Resumability によるパフォーマンス最適化

## 概要

**Qwik** は、エッジ向けに再考されたフロントエンドフレームワーク。ハイドレーション処理をスキップし、瞬時にインタラクティブなWebアプリケーションを実現することを目指す。

### 主な特徴

- **Resumability（再開可能性）** - ハイドレーション不要で、即座にインタラクティブ化
- **パフォーマンス最適化** - インスタント読み込み、レンダリング削減、スケーラブル設計
- **開発者体験** - React風のJSX構文、学習コストが低い

---

## Resumability（再開可能性）の詳細

### コア概念

**Resumability = サーバーで実行を一時停止 → クライアントで再開**

従来のSSR/SSGと異なり、全コンポーネントのリダウンロードと再実行（ハイドレーション）が不要。

### ハイドレーション vs Resumability

**従来のフレームワーク（React等）のハイドレーション:**

```
サーバー: HTMLを生成
         ↓
クライアント:
  1. イベントリスナーをDOMに設置
  2. コンポーネントツリー（内部状態）を再構築
  3. アプリケーション状態を復元
  ※ すべてのコンポーネントコードをダウンロード→テンプレート実行
```

問題点：このプロセスのオーバーヘッドが大きい（Time to Interactive: 3-5秒）

**QwikのResumability:**

```
サーバー: 実行状態を一時停止してシリアライズ → HTMLに埋め込み
         ↓
クライアント: シリアライズされた状態から直接再開
             → コンポーネント再実行不要（TTI: <100ms）
```

### 3つの実装メカニズム

#### 1. イベントリスナーのシリアライゼーション

```html
<button on:click="./chunk.js#handler_symbol">click me</button>
```

- リスナーをDOM属性に埋め込み
- Qwikloaderが単一のグローバルリスナーで全イベント捕捉
- 必要なチャンク・関数だけ遅延読み込み

#### 2. コンポーネントツリー情報のシリアライゼーション

- SSR/SSG時にコンポーネント境界をHTMLに記録
- クライアントはコンポーネントコードなしで階層構造を再構築
- 必要なコンポーネントのみ段階的に読み込み

#### 3. アプリケーション状態の分離管理

- 親コンポーネント無しに個別コンポーネントを復元可能
- 状態はコンポーネントライフサイクルに統合
- 細粒度なダウンロード・再実行が可能

### プログラミングパラダイム的背景

**重要な転換：「ヒープ中心」→ 「DOM中心」**

従来のフレームワークは状態をメモリ（ヒープ）に保持し、コンポーネント中心の設計。
Qwikは状態をDOMに埋め込み、DOM中心の思考パラダイムが必要。

Qwikの利点を活かすには、開発者がこの新しい設計思想を採用する必要がある。

## Resumability と Coroutine/Generator の類似性

### 共通点

```javascript
// Generator的な考え方
function* computation() {
  const x = expensive_calc();  // ← 状態A
  yield x;
  // ← ここから再開、xを持った状態で続行
  const y = another_calc(x);
  return y;
}

// Qwik Resumabilityの考え方（概念的に）
// サーバー側で状態をスナップショット、クライアント側で再開
```

どちらも「関数の実行状態を保存し、後で状態から復帰して実行を続ける」パターン。

### 本質的な違い

| 項目 | Generator/Coroutine | Qwik Resumability |
|------|-------------------|------------------|
| 実行環境 | 同一プロセス内 | サーバー ⇄ クライアント（プロセス跨ぎ） |
| 状態保存 | スタック・レジスタ | JSON的シリアライゼーション |
| 時間スケール | ミリ秒単位 | ネットワーク遅延（秒単位） |
| 制御フロー | 線形的・確定的 | イベント駆動・非確定的 |

### より正確なアナロジー

Qwikは **「Continuation-Passing Style（CPS）」** や **「Delimited Continuations」** の概念に近い。

プログラミング言語的背景：
1. **Functional Programming** - 純粋関数、イミュータビリティ
2. **Lazy Evaluation** - 必要な時だけ実行
3. **Continuation** - 実行を一時停止して中断・再開

これらの組み合わせで、サーバー・クライアント間での状態引き継ぎを実現。

## Resumability がなぜパフォーマンス向上に繋がるのか

### 初期化するコードが圧倒的に少ない

**React（ハイドレーション）:**
```javascript
React.render(
  <App />,  // ← 全コンポーネントのコードが必要
  document.getElementById('root')
);
// バンドルサイズ: ~300KB
```

**Qwik:**
```javascript
QwikLoader.init();  // ← これだけ
// バンドルサイズ: ~10KB
```

### 遅延読み込みの粒度が細かい

**React的:** ユーザー操作 → ページ全体のコンポーネント読み込み（不要なコンポーネントもロード）

**Qwik的:** ユーザー操作 → 該当イベントハンドラーだけロード

例：
- クリックしたボタン → 1-2KB
- 別のボタンをクリック → 別の 1-2KB
- 触らなかったボタン → 0KB

### State/Closureがシリアライズされている

```javascript
// HTMLに埋め込まれた状態
<button 
  on:click="./handlers.js#handleClick"
  data-closure="eyJjb3VudCI6IDAsIC4uLn0="  // ← Base64化した状態
>
  Click me
</button>

// クライアント側：既にクロージャが完成した状態で再開
// useState/useEffect の再実行不要
```

### 処理時系列の違い

```
React（ハイドレーション）:
Timeline: 0ms ─────────── 500ms ──────────── 1500ms ─── 3000ms
           HTML受け取り    JS DL   ハイドレーション実行   TTI
           見える        ⏳      CPU高負荷 ⏳⏳

Qwik（Resumability）:
Timeline: 0ms ─────── 50ms ──── ユーザー操作 ──── 150ms
           HTML受け取り  TTI   JS DL（必要分）   イベント実行
           見える ✓  インタラクティブ ✓        ✓
```

### React Server Components との比較

| 項目 | React + RSC | Qwik |
|------|-----------|------|
| バンドル戦略 | サーバー/クライアント分離 | 状態/参照をHTML埋め込み、コード遅延読み込み |
| 初期化コード | クライアント側コンポーネント全部 | Qwikloaderのみ |
| **粒度** | **コンポーネント単位** | **イベントハンドラー単位** ← ここが重要！ |
| State管理 | useStateを再実行 | シリアライズされた状態から直接再開 |
| TTI | 1-2秒程度 | <100ms |

**RSCの限界:** サーバー側の処理をサーバーに留めても、クライアント側のコンポーネント（useState/useEffect持つもの）はまだ全部バンドルして評価する必要がある。

**Qwikの利点:** 評価（evaluation）そのものをスキップ。シリアライズされた状態から直接実行を再開。

---

## 本質的なメリット

1. **JavaScript評価のコスト削減** - 実行済み状態から再開だから、大量の関数評価をスキップ
2. **必要分だけロード** - ページ全体ではなく、イベントごとに必要なコード片だけダウンロード
3. **初期化のオーバーヘッド削減** - React全体の再構築が不要

---

## その他の利点

- PWAアプリの状態保存・復元
- レンダリング性能向上（変更コンポーネントのみ再描画）
- 細粒度の遅延読み込み
- モバイルデバイスのメモリ圧力軽減
- 既存静的サイトの段階的インタラクティブ化

### シリアライゼーション形式のサポート

サポート：
- 循環参照
- DOM参照
- Date、Promise、URL、Map、Set など

制限：
- クラスのシリアライゼーション（instanceof/prototype）
- ストリーム処理
