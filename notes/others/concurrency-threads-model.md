## スレッドモデルの分類

### グリーンスレッド（M:1、古い概念）

- JVM や言語ランタイムが管理する軽量スレッド
- 多数のグリーンスレッドを **1つの OS スレッド** にマップ
- シングルコアしか使えない → マルチコア CPU を活かせない
- Java 1.1〜1.2 時代に存在したが廃止

### 仮想スレッド・Goroutine（M:N、現代）

- 多数の軽量スレッドを **複数の OS スレッド（キャリアスレッド）** にマップ
- マルチコアをフル活用できる
- I/O ブロッキング時にキャリアスレッドを解放し、別の軽量スレッドを実行

```
仮想スレッド/Goroutine (数百万個)
       ↓ ランタイムがスケジューリング
  OS-T  OS-T  OS-T  OS-T  ← CPU コア数分
```

### Node.js イベントループ（どちらとも違う）

- シングルスレッド + イベントループによる非同期 I/O
- スレッドを複数作らず、1スレッドで非同期コールバックを順番に処理
- I/O は OS の非同期機能（epoll/kqueue）に委譲
- `async/await`、callback で表現

```
コールバックキュー → イベントループ → 実行 → 次のコールバック
                          ↓
                     OS の非同期 I/O
```

## 各技術の比較

| | モデル | マルチコア | 書き方 |
|---|---|---|---|
| グリーンスレッド | M:1 | 非対応 | スレッド API |
| Node.js イベントループ | 1スレッド + 非同期 | 非対応※ | async/await, callback |
| Go Goroutine | M:N | 対応 | 同期的に書ける |
| Java 仮想スレッド（Java 21〜） | M:N | 対応 | 同期的に書ける |

※ `worker_threads` や `cluster` でマルチコアは可能だが、別プロセス/スレッドを立てる話

## Java 仮想スレッドの使い方

```java
// 仮想スレッドで実行
Thread.ofVirtual().start(() -> {
    // I/O などのブロッキング処理
});

// ExecutorService 経由
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> doWork());
}
```

**注意点：**
- CPU バウンドな処理には向かない（I/O バウンドに真価を発揮）
- `synchronized` ブロック内でブロックするとキャリアスレッドが固定（ピン留め問題）→ Java 23 以降で改善

## プリエンプティブ vs コオペレイティブ

### プリエンプティブ（Preemptive）

**スケジューラが強制的に切り替える**方式。

- タイマー割り込みなどで CPU を奪う
- タスク側は「いつ切り替わるか」を意識しなくていい
- OS のスレッドスケジューリングがこれ

```
タスクA 実行中 → [タイマー割り込み] → スケジューラが強制停止 → タスクB へ
```

### コオペレイティブ（Cooperative）

**タスク自身が明示的に制御を手放す**方式。

- `yield` や `await` など、タスクが「ここで切り替えていいよ」と宣言する
- タスクが CPU を手放さないと他のタスクが動けない
- 1つのタスクが暴走すると全体が止まる

```
タスクA 実行中 → [await/yield] → 自発的に手放す → タスクB へ
```

### スケジューリング方式の対応

| | 方式 |
|---|---|
| OS スレッド | プリエンプティブ |
| Go Goroutine | ハイブリッド（Go 1.14〜 基本プリエンプティブ） |
| Java 仮想スレッド | ハイブリッド（基本プリエンプティブ） |
| Node.js / async-await | コオペレイティブ |
| Python asyncio | コオペレイティブ |
| 古いグリーンスレッド | コオペレイティブが多かった |

### コオペレイティブの問題点

```python
async def bad_task():
    while True:
        heavy_cpu_work()  # await しないと永遠に制御を返さない
        # 他のタスクが一切動けない
```

これが async/await を使う言語で **CPU バウンド処理が苦手**な理由。

## まとめ

- **グリーンスレッド**：M:1、シングルコアのみ、歴史的遺物
- **仮想スレッド / Goroutine**：M:N、マルチコア対応、同期的に書けてスケールする
- **イベントループ**：スレッドではなくコールバックで非同期を表現、async/await が必要
- **プリエンプティブ**：スケジューラが偉い、タスクは何も考えなくていい
- **コオペレイティブ**：タスクが協調して動く、暴走リスクあり
- 現代の M:N スレッドはプリエンプティブ寄りのハイブリッドで両方の良いとこ取り
